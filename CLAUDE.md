# Burlington Consult — Technical Architecture & Code Conventions

**Purpose:** This is the technical rulebook for the Burlington Consult web application. Paste this into your Claude Code `CLAUDE.md` at the root of the monorepo. It tells Claude Code how to write code for this project.

**Companion document:** `burlington-webapp-build-plan.md` (the sprint-by-sprint plan)

---

## TL;DR — If You Read Nothing Else, Read This

These are the absolute non-negotiables. Every file you write must comply with these. If anything below this TL;DR contradicts these rules, these win.

1. **RLS on every table.** Row-Level Security is enabled and policies are written in the same migration that creates the table. No exceptions.
2. **Zod validates every input.** Form submissions, API payloads, webhook bodies, env vars. If it crosses a boundary, it's validated.
3. **Money in integer cents.** Never floats, never decimals. USD in cents, NGN in kobo. Conversion happens at the display layer only.
4. **Configurable, not hardcoded.** Prices, team members, pricing tiers, feature flags, country lists, currency activations, copy snippets, stats — all admin-editable through the Configuration System. If Chris might plausibly want to change it without a developer, it's config, not code. See Admin Configuration System.
5. **Server Components by default.** Only mark `'use client'` when you genuinely need state, effects, event handlers, or browser APIs.
6. **Never bypass auth.** Every protected page calls `requireUser()`. Every protected Server Action re-checks auth. Don't assume.
7. **Service role key is server-only.** Never expose to the browser. Never use it except in trusted server code (webhooks, cron jobs, admin-only mutations).
8. **Errors are handled, not swallowed.** Every async operation has error handling. Every error is logged with context. Every user-facing failure has a clear message.
9. **Loading, empty, error states on every data fetch.** No screen ever blanks out or hangs. The user always knows what's happening.
10. **Webhooks are idempotent.** Every webhook handler deduplicates by provider event ID before processing. Retries are expected.
11. **No inline styles for static values.** Tailwind utilities for everything except runtime-computed values (progress widths, DB-driven colors, etc.).
12. **International by default, never Nigeria-by-assumption.** A user may be Nigerian, Indian, British, American, Brazilian, Ghanaian — the code cannot assume. See International by Default.

If you're about to write code that violates any of these, stop and re-read the relevant section.

## Attribution Rules

**Claude is a tool Adonai uses privately. Claude is never a contributor, collaborator, or author.** Every public-facing output — commits, PR descriptions, code comments, documentation, emails, error messages, about pages, footers, legal documents — must be authored as if Adonai wrote it.

### What this means in practice

**Commit messages:**
- Write in first person or imperative mood ("add booking form", "fix NDA signature validation")
- Author: Adonai Jonathan (git config on his machine)
- **Never** append `Co-authored-by: Claude`, `Generated with Claude Code`, or any equivalent
- **Never** reference AI assistance in the commit body

**PR descriptions:**
- Written as Adonai would write them: what changed, why, testing notes, screenshots
- No AI disclosure
- No "generated by" or "assisted by" attribution

**Code comments:**
- Explain the why, not the who
- No `// Claude suggested this` or equivalent
- If an approach is non-obvious, document the reasoning — anonymously

**Documentation (READMEs, public docs, release notes):**
- Credit human contributors only
- Sample author block: "Built by Adonai Jonathan. Designed by Stanley Ojima."
- No AI mentions in changelogs, release notes, or "About" sections

**In-product text (emails, UI copy, legal templates):**
- Written in Burlington's voice, not Claude's
- No meta-commentary about how content was generated
- No "our AI assistant helped draft this" anywhere

**Error messages, status pages, admin UIs:**
- "Something went wrong. We've been notified." — not "Claude encountered an error."
- Sentry error reports are internal; even there, no Claude attribution needed

### Why this rule exists

The distinction is real: Claude is a tool, like an IDE, like Prettier, like Vercel. None of those get attribution in commit history. Claude shouldn't either. The work is Adonai's. The product is Burlington's. Internal tooling choices are internal.

### What's fine

- `CLAUDE.md` itself — this file lives at the repo root and is read by Claude Code as a convention. It's a configuration file, not a public document. It can reference Claude Code freely because that's its purpose.
- Internal Slack / private conversations about the engineering process
- Anything in `.gitignore` or private notes

### What's not fine

- Anything in the git log accessible via `git log`
- Anything in the deployed product
- Anything a journalist, investor, client, or competitor could view
- Any PR, issue, discussion, or commit description on GitHub

**The test:** If you're about to write text that will be visible to anyone outside Adonai's own private notes, scan it for "Claude", "AI", "generated", "assisted", "co-authored". If any of those appear attributing the work, rewrite.

## International by Default

**Burlington is a global firm that happens to be based in Lagos and Delaware.** Not a Nigerian firm that happens to serve some international clients. This distinction shapes every design and code decision in this project.

The platform serves professionals in Nigeria, the U.S., the U.K., Canada, India, Pakistan, Ghana, Kenya, South Africa, the Philippines, China, Brazil, Germany, France, Australia, and everywhere else where ambitious professionals file for EB-1A or EB-2 NIW. A Nigerian doctor living in Canada, an Indian researcher in Singapore, a British executive in Dubai — all are equally Burlington clients. None should feel like afterthoughts.

### The principle

If a piece of code, copy, or UI assumes the user is Nigerian, it's wrong. If it assumes the user is in the U.S., it's also wrong. The platform must feel native to a professional anywhere in the world — Nigerian clients should feel first-class, and international clients should feel first-class. The infrastructure is built for Nigeria (Paystack) and the world (Stripe, NOWPayments) simultaneously, and the UX must match.

### What this means in practice

**Never assume citizenship from location.** A Nigerian passport holder living in Canada pays from Canada. A British citizen living in Lagos pays in Naira. Citizenship, residence, and payment location are three separate fields.

**Never assume location from language.** English is the lingua franca of EB-1A filers globally. Don't code-switch assuming the user is Nigerian, American, or British.

**Never default to "United States" or "Nigeria" in country selectors.** Both are bias. Either auto-detect from IP (and let the user override) or present an empty placeholder ("Select country").

**Country lists are comprehensive.** Not a curated list of 15 countries with "Other" at the bottom. Use the full ISO 3166-1 list. If this feels excessive, the trimmed list you're considering is biased.

**Timezone-aware everywhere.** The client sees everything in their timezone. The adviser sees everything in theirs. Emails render times in the recipient's timezone. Stored timestamps are UTC. See Dates, Times & Timezones section.

**Currency activation is admin-gated.** The platform supports NGN, USD, GHS, KES, ZAR, and others architecturally, but each non-core currency is disabled by default and activated by a super admin when Burlington is ready to serve that market. At launch, only NGN and USD are active — Ghanaian, Kenyan, and South African clients see USD until those currencies are explicitly switched on in admin settings. This keeps the infrastructure ready without forcing pretend-support. See Admin Configuration System below.

**Currency conversion is live and visible where activated.** A Nigerian client seeing a $20,000 engagement sees ₦32,500,000 equivalent with the exchange rate and timestamp. Once GHS activation flips on, Ghanaian clients see GH₵ equivalents the same way. The underlying charge is in the active currency that matches their payment method, not an arbitrary default.

**Names respect cultural variation.** Not every culture has First Name + Last Name. Some have one name. Some put family name first. Some have multiple components. Use one `full_name` field as the primary, optionally collect `given_name` and `family_name` separately when needed (e.g., for legal documents). Never force a parse.

**Addresses are flexible.** Don't hardcode "State" and "ZIP code" — those are U.S. conventions. Use `address_line_1`, `address_line_2`, `city`, `administrative_area` (state/province/county/region), `postal_code`, `country_iso`. Let country-specific validation rules live in a helper, not the schema.

**Phone numbers use E.164 with a country code picker.** Never `+234-***` as placeholder. Use `libphonenumber-js` for parsing and formatting. Store `+234XXXXXXXXXX` not `0XXXXXXXXXX`.

**Dates display per user locale.** Americans read MM/DD/YYYY. Most of the world reads DD/MM/YYYY. ISO format (YYYY-MM-DD) is universal and preferred in system contexts. Display format uses `Intl.DateTimeFormat` with the user's browser locale — not a hardcoded string.

**Content tone is internationally neutral.** No "we Nigerians," no "the U.S.," no "our country." Burlington speaks as a global firm. References to Nigeria or the U.S. are specific ("our Lagos office," "clients filing to U.S. USCIS"), not generic. British English spelling is the default (matches the existing marketing site).

**Team positioning is expertise-first, geography-second.** The team is largely Nigerian, which is a source of strength, not something to downplay or overemphasize. Credentials (Harvard Law, Harvard Business School, Cambridge, Rotman, PwC, Meta, Microsoft, Luminate) do the positioning work. Burlington is not "the Nigerian EB-1A firm" — it's the global EB-1A firm with a Nigerian-led team.

**Case profiles represent diverse geographies.** The Cases page cannot be all Nigerian professionals. It needs representative anonymised profiles from multiple countries and fields — Indian tech, British finance, Brazilian researchers, Kenyan policy, alongside Nigerian law and medicine. Reflects the actual client base the firm wants to serve.

**Legal disclaimers work in every jurisdiction where the platform is used.** "Burlington Consult does not practise law" is universal. Specific regulatory disclaimers (state bar rules in the U.S., SRA rules in the U.K., etc.) are not our concern because we don't practise law in any of them — but the disclaimer copy must never imply we do.

**Privacy compliance is multi-jurisdictional.** See Privacy & Compliance section below. GDPR for EU visitors, UK GDPR for British visitors, CCPA/CPRA for Californians, NDPR for Nigerians, LGPD for Brazilians, PIPEDA for Canadians. The simplest path is to meet the strictest standard (GDPR) everywhere — which is the default for this build.

### Where Nigeria-specific logic is appropriate

Nigeria-specific logic is appropriate when it's about infrastructure, not identity:

- Paystack as the payment processor for clients paying in NGN (Nigerian card rails, capped fees)
- Naira display format (`₦32,500,000.00`) when the selected currency is NGN
- NDPR compliance for Nigerian residents
- A Nigerian phone number country code (+234) in the country picker, same as every other country code

Nigeria-specific logic is NOT appropriate when it assumes the user:

- "Based on your Nigerian profile..." ❌ — the user may not be Nigerian
- Defaulting WhatsApp as the primary contact method — WhatsApp is strong in Nigeria, LATAM, and India but not in the U.S., U.K., or Germany
- Defaulting Nigerian English conventions in copy — use British English spelling (`endeavour`, `organisation`, `recognised`) as an internationally neutral register, not American English (`endeavor`, `organization`) which reads as US-centric

## Never Do

A single consolidated list of anti-patterns. When you're about to write something, scan this list first.

**Security**
- Never use `SUPABASE_SERVICE_ROLE_KEY` in client code or in any file that might get bundled for the browser
- Never bypass RLS by "just using the service role for this one thing" in a user-facing request path
- Never trust client-supplied user IDs — always use the authenticated session's user ID
- Never log PII, tokens, full card numbers, signed NDAs, or passwords (log opaque IDs only)
- Never skip webhook signature verification
- Never commit `.env.local`, `.env.production`, or any file containing real keys

**TypeScript**
- Never use `as any` (use `unknown` and narrow with Zod, or fix the type)
- Never use `@ts-ignore` (use `@ts-expect-error` with a comment explaining why, if absolutely necessary)
- Never use `!` non-null assertions in app code (narrow the type or handle the null case)
- Never disable ESLint rules without an inline comment explaining why
- Never export `enum` — use string literal union types (`type Status = 'pending' | 'approved'`)

**Data & Money**
- Never store money as `decimal`, `float`, `real`, or `numeric` with scale — always `integer` cents
- Never do arithmetic on different currencies in the same expression without explicit conversion through a rates service
- Never use `new Date()` directly inside business logic — go through the clock service so tests can mock time
- Never store timestamps as `timestamp` — always `timestamptz`
- Never use Postgres `enum` types — use `text` with check constraints (easier to evolve)
- Never reconstruct a refund amount — refund the exact amount that was charged, pulled from the original payment record

**Next.js & React**
- Never use `useEffect` to fetch data in a page (use Server Components or React Query in client components)
- Never mix server and client code in the same file (split into two files, one explicitly `'use client'`)
- Never use `<form action={...}>` with a URL in Server Components unless you specifically want a page navigation — use Server Actions
- Never use `<div onClick>` — use `<button>` (it's keyboard-accessible and semantic)
- Never mutate props or state directly (React assumes immutability)
- Never rely on `localStorage` or `sessionStorage` in Server Components (they don't exist server-side)

**Styling**
- Never use `@apply` in CSS files (use utilities directly, or extract to a component with `cva`)
- Never use inline styles for static values (see the Tailwind section for the dynamic-value exception)
- Never use CSS modules, styled-components, emotion, or any other CSS-in-JS library — Tailwind only
- Never remove default focus outlines without providing an equivalent (accessibility)

**Forms & Validation**
- Never submit a form without Zod validation (both client-side for UX and server-side for trust)
- Never hardcode user-facing strings inside validation logic — the error message belongs in the schema
- Never trust form data in a Server Action without re-parsing with Zod

**Database**
- Never write migrations that `DROP` tables or columns in production (use soft deletes or new migrations to alter)
- Never skip RLS policies on a new table, even temporarily
- Never write a query without considering the index strategy (every foreign key, every WHERE column used in production)
- Never run raw SQL from client-side code (always through server functions with validated inputs)

**Files & Uploads**
- Never trust the client's claimed MIME type — detect server-side from magic bytes or at minimum validate extension against MIME
- Never use a user-supplied filename as the storage path (use `{userId}/{timestamp}-{slug}.{ext}`)
- Never expose public URLs for private files — use signed URLs with short TTLs
- Never accept an upload without a size limit

**Configuration & Content**
- Never hardcode a price, fee, percentage, or monetary threshold in the codebase — all monetary values come from `app_config` or the relevant product tables
- Never hardcode a team member's name, title, email, or bio anywhere — team data lives in the `profiles` + `team_profiles` tables
- Never hardcode user-facing copy that might plausibly change (homepage hero, email subject lines, stat values, legal clauses) — store in `app_config` with a stable key
- Never hardcode a list of countries, currencies, or payment providers — these are config surfaces the admin must control
- Never implement a new feature without considering whether its key parameters should be admin-editable — the default is yes, not no
- Never apply a config change without going through the approval/audit flow — bypassing it breaks the trust model Chris relies on

**External APIs & Resilience**
- Never call an external API without an explicit timeout — a hung request eventually takes down the worker
- Never retry non-idempotent POST requests automatically (could double-charge, double-email, double-activate)
- Never ignore a 429 rate-limit response — respect `Retry-After` header
- Never hammer a failing service — use circuit breakers for critical external dependencies
- Never show raw third-party error messages to users — translate to something actionable

**Rate Limiting & Abuse**
- Never expose a public endpoint without rate limiting (especially auth, apply, chat, assessment)
- Never rate limit by user ID on endpoints that are pre-authentication — use IP instead
- Never return a 429 without a `Retry-After` header — clients need to know when to come back

**Headers & CORS**
- Never use `Access-Control-Allow-Origin: *` with credentials — always explicit origins
- Never deploy to production without CSP in enforce mode (not report-only)
- Never bypass CSP by adding blanket `'unsafe-inline'` without documenting why

**Forms & Data**
- Never let a multi-step form lose data on network drop — persist drafts to localStorage
- Never return unbounded query results — always paginate
- Never hard-delete user-facing records in regular operations — soft delete with `deleted_at`
- Never drop a column in the same deploy as the code change using it — expand/contract pattern only

**Realtime**
- Never forget to clean up Supabase Realtime subscriptions on unmount (memory leak + ghost updates)
- Never broadcast typing indicators on every keystroke — throttle to 1.5s minimum

**Dependencies**
- Never install a new package without checking bundle size and existing equivalents
- Never install packages with GPL or AGPL licenses
- Never pin exact versions in `package.json` (use caret ranges) unless there's a known breaking change in a minor version

**Git & Deploy**
- Never `git push --force` to `main` or any shared branch
- Never deploy to production without a preview deploy + human review
- Never commit commented-out code blocks ("might need this later" is what git history is for)
- Never commit `console.log` statements that shipped from debugging
- Never attribute work to Claude, AI, or any generation tool in commits, PR descriptions, or public docs (see Attribution Rules)
- Never add `Co-authored-by: Claude` or `Generated with Claude Code` to commit messages
- Never include "AI-generated", "AI-assisted", "generated by" in README files, release notes, or changelogs

### Payments
Four providers, all first-class at launch. Client sees the right one based on location and preference. Webapp treats them uniformly at the data layer (all payments normalize to the same `payments` table).

- **Nigeria (cards, bank, USSD):** Paystack (Standard API). Fee: 1.5% capped at ~$1.25. Default for Nigerian IPs.
- **Ghana, Kenya, South Africa (cards, bank, mobile money):** Flutterwave. Fee: 1.4% local / 3.8% international. Default for these African markets.
- **Rest of the world (cards):** Stripe (Checkout Sessions via API, not Payment Links). Fee: 2.9% + $0.30. Default for all other IPs.
- **Stablecoin (USDC, USDT):** NOWPayments. Fee: 0.5% + 0.5% auto-conversion to USD = 1% total. Non-custodial — funds route to Burlington's wallet. Always visible as an alternative payment method regardless of location.

**Currency detection at checkout:**
- Nigerian IP → defaults to Paystack, displays prices in NGN with USD reference
- Ghanaian IP → Flutterwave, GHS with USD reference
- Kenyan IP → Flutterwave, KES with USD reference
- South African IP → Flutterwave, ZAR with USD reference
- All other IPs → Stripe, USD
- Stablecoin option always visible as an alternative tab
- Client can override any default via a currency/method selector

**Pricing principle — one USD reference price, shown in local currency at checkout:**

Burlington has one global price for each product ($500 assessment, $1,000 founder consultation, $20K engagement, etc.). There is no purchasing-power-parity discount. What changes by region is the *display* and *payment rail*, never the price itself.

A Nigerian client sees ₦820,000 at checkout *with* "($500 USD)" visible alongside. The exchange rate and timestamp are shown. The underlying charge is in NGN on Paystack. A Ghanaian client sees GHS, charged in GHS on Flutterwave. A US client sees $500, charged in USD on Stripe. All three paid the same price in USD terms; they just saw it and paid it in whatever way their region makes easy.

**Key integration rule:** Every payment, regardless of provider, creates one row in `payments` with `provider`, `amount_usd`, `amount_local`, and `currency` fields set. The USD equivalent is always computed and stored, so revenue reporting works across all providers in one currency.

**First-class African markets at launch:** Nigeria, Ghana, Kenya, South Africa. These are priority markets with dedicated payment rails, local currency display, and testing pre-launch. Other African countries work but fall back to Stripe — acceptable for launch, optimise post-launch based on demand signal.

### Deployment
- **Frontend + API routes:** Vercel (Pro tier) — hosts both Next.js apps (portal + admin), runs serverless functions for short-lived backend work (webhooks, form submissions, checkout sessions)
- **Background worker:** Railway — runs a persistent Node.js service for anything that doesn't fit in Vercel's 60-second serverless limit (scheduled jobs, batch processing, long-running tasks)
- **Database + storage + auth:** Supabase Pro

- **DNS:** Cloudflare

**Three-service architecture:**

```
User → Cloudflare → Vercel (Next.js apps)
                        ↓ writes / reads
                    Supabase (Postgres + Auth + Storage + Realtime)
                        ↑ polls / reads
                    Railway (Node.js worker) → sends emails, runs cron, processes queued jobs
```

**What runs where:**

| Service | Runs on | Example work |
|---|---|---|
| Web pages, forms, auth flows | Vercel | Client loads dashboard, submits application, signs NDA |
| Webhook handlers | Vercel | Stripe payment success, Paystack charge success, NOWPayments IPN |
| Server Actions (form mutations) | Vercel | Create booking, submit application, upload document |
| Chatbot proxy | Vercel | Route user message to chat service, return reply |
| Scheduled jobs | Railway | Nightly safelock inactivity check, monthly reports |
| Batch email sends | Railway | Consultation reminders 24h before, weekly digest |
| Long-running PDF generation | Railway | Bulk engagement letter generation |
| Queue processing | Railway | Poll `pending_jobs` table, execute, mark complete |

**Why split:** Vercel serverless functions timeout at 60 seconds and spin up fresh per request. Some work (scanning thousands of rows, sending batches of emails, generating many PDFs) doesn't fit that model. Railway runs a persistent Node.js service that's always on, ideal for background work.

## Architecture Principles

### 1. Server-first

**Default to Server Components.** Only mark a component `'use client'` when it genuinely needs:
- `useState` / `useEffect` / `useReducer`
- Event handlers (`onClick`, `onChange`, `onSubmit`)
- Browser-only APIs (`window`, `localStorage`, `document`)
- Third-party libraries that require client execution

When in doubt, start as a Server Component. Move the interactive leaf to a small client component and pass data down.

**Rule of thumb:** A page should be a Server Component that fetches data and renders. Interactive widgets inside the page are client components.

### 2. Co-locate everything

Component folder structure:
```
components/
  booking/
    BookingForm.tsx              # The main component
    BookingForm.test.tsx         # Tests
    TierSelector.tsx             # Sub-component
    useAvailability.ts           # Local hook
    schema.ts                    # Zod schema
    types.ts                     # Local types
    index.ts                     # Public exports
```

Feature lives in one folder. No `types/`, `hooks/`, `utils/` at the root — those are for truly shared things.

### 3. Separate concerns at boundaries

| Layer | Purpose | Example |
|---|---|---|
| **Page** (`app/**/page.tsx`) | Fetch data, compose UI | `app/(dashboard)/book/page.tsx` |
| **Server Action** (`actions/`) | Write operations from forms | `createBooking(formData)` |
| **API Route** (`app/api/`) | Webhooks, external callbacks | `app/api/webhooks/stripe/route.ts` |
| **Data layer** (`lib/db/`) | Database queries | `lib/db/bookings.ts` |
| **Business logic** (`lib/services/`) | Non-CRUD logic | `lib/services/availability.ts` |
| **UI components** (`components/`) | Presentation | `components/booking/BookingForm.tsx` |

Never mix these. A page doesn't call Supabase directly — it calls a function in `lib/db/`. A component doesn't call Stripe directly — it triggers a Server Action that calls `lib/services/stripe.ts`.

### 4. Type safety end-to-end

Types flow from one source: the Supabase schema.

```ts
// packages/shared/supabase/types.ts (auto-generated from Supabase CLI)
export type Database = { /* ... */ }

// packages/shared/types/index.ts
import type { Database } from '../supabase/types'

export type Profile = Database['public']['Tables']['profiles']['Row']
export type Application = Database['public']['Tables']['applications']['Row']
// etc.
```

Run `pnpm db:types` after every schema migration to regenerate. Commit the generated file.

### 5. Security by default

Every table has Row-Level Security (RLS) enabled. Every policy is tested. No query uses the service role key except in trusted server-side code (webhooks, cron jobs, admin mutations).

**RLS policy conventions:**
- Clients read/write their own rows only
- Advisers read rows for their assigned clients
- Admins read own + assigned clients' rows
- Super admins read all rows

Write RLS policies in migration files, never manually in the dashboard.

## Code Style

### TypeScript

**Strict mode on. Inference where clean.**

```ts
// ✅ GOOD — inference is obvious
const user = await getUser(id)
const emails = users.map(u => u.email)

// ✅ GOOD — explicit where it matters
export async function createBooking(input: CreateBookingInput): Promise<Booking> {
  // ...
}

export interface BookingFormProps {
  adviserId: string
  tier: AdviserTier
  onSuccess?: (booking: Booking) => void
}

// ❌ BAD — unnecessary annotation
const name: string = user.fullName
const count: number = 5

// ❌ BAD — `any` is banned except in explicit escape hatches
function doThing(data: any) { }

// ✅ ESCAPE HATCH — when you must, comment why
// eslint-disable-next-line @typescript-eslint/no-explicit-any -- Third-party lib lacks types
function webhookHandler(payload: any) { }
```

**Rules:**
- Explicit types on function parameters (always)
- Explicit return types on exported functions (always)
- Infer return types on internal/local functions (usually fine)
- Explicit types on React component props (always — use an `interface`)
- No `any`. Use `unknown` and narrow with Zod.
- Prefer `type` for unions/primitives, `interface` for object shapes and component props
- Enums: use string literal union types instead (`type Status = 'pending' | 'approved'`)

### Naming

| Thing | Convention | Example |
|---|---|---|
| Files — Components | PascalCase | `BookingForm.tsx` |
| Files — Hooks | camelCase, `use` prefix | `useAvailability.ts` |
| Files — Utilities | kebab-case | `format-currency.ts` |
| Files — Routes | kebab-case | `app/(dashboard)/book-consultation/` |
| Variables & functions | camelCase | `const userEmail`, `function getBooking()` |
| Types & interfaces | PascalCase | `type BookingStatus`, `interface UserProfile` |
| Constants (true constants) | SCREAMING_SNAKE_CASE | `const MAX_UPLOAD_SIZE = 10_000_000` |
| Enums (if you must) | PascalCase | `PaymentStatus.Pending` |
| Database columns | snake_case | `created_at`, `full_name` |
| Env vars | SCREAMING_SNAKE_CASE | `STRIPE_SECRET_KEY` |

**Booleans:** prefix with `is`, `has`, `should`, `can` — `isLoading`, `hasError`, `canEdit`

**Event handlers:** prefix with `on` for props, `handle` for implementations
```tsx
<Button onClick={handleSubmit} />
```

### Tailwind & Styling

**Use Tailwind utility classes. No CSS modules. No styled-components. No `@apply` in CSS files.**

**On inline styles — the rule:**

> Inline styles (`style={{ ... }}`) are banned for all static values. Use Tailwind utilities. Inline styles are permitted ONLY for dynamic values that cannot be expressed as utility classes at build time.

**What counts as a dynamic value (inline style permitted):**

```tsx
// ✅ OK — progress bar width computed from data
<div
  className="h-2 rounded-full bg-teal transition-all"
  style={{ width: `${(deposited / target) * 100}%` }}
/>

// ✅ OK — adviser profile theme color from DB
<div
  className="rounded-lg border p-4"
  style={{ borderColor: adviser.themeColor }}
/>

// ✅ OK — chart dimensions computed from container size
<svg style={{ width: containerWidth, height: containerHeight }}>
  {/* ... */}
</svg>

// ✅ OK — CSS custom properties for theming a section
<section style={{ '--accent-color': palette.accent } as React.CSSProperties}>
  {/* ... */}
</section>

// ✅ OK — transform computed from drag position
<div style={{ transform: `translate(${x}px, ${y}px)` }} />
```

**What's banned (use Tailwind instead):**

```tsx
// ❌ BAD — static values belong in utilities
<div style={{ padding: '16px', color: '#0D9488' }}>

// ✅ GOOD
<div className="p-4 text-teal">

// ❌ BAD — simulating hover with inline styles
<div style={{ backgroundColor: isHovered ? '#0D9488' : 'transparent' }}>

// ✅ GOOD — use Tailwind state modifier
<div className="bg-transparent hover:bg-teal">

// ❌ BAD — conditional classes via inline style
<div style={{ display: isOpen ? 'block' : 'none' }}>

// ✅ GOOD — conditional className
<div className={isOpen ? 'block' : 'hidden'}>
// or better: { !isOpen ? null : <Component /> }
```

**Rule of thumb:** If the value is known at build time (a color from the palette, a spacing value, a font size), it's Tailwind. If the value is computed at runtime from state, props, or DB data, inline is acceptable.

**When in doubt, try Tailwind's arbitrary value syntax first:**

```tsx
// Sometimes arbitrary values work and keep everything in className
<div className="w-[calc(100%-2rem)] bg-[#0D9488]/80">
```

### Multi-step form persistence

Long forms (application, NDA, onboarding, engagement letter) must survive network drops, browser crashes, and accidental tab closures. Nothing is worse than losing 20 minutes of thoughtful answers because wifi blipped.

**Rule: any form with more than 4 fields, or any form taking > 2 minutes to complete, persists to localStorage on every change.**

```tsx
'use client'

import { useEffect } from 'react'
import { useForm } from 'react-hook-form'

const STORAGE_KEY = 'burlington:form:application'

export function ApplicationForm() {
  const { register, handleSubmit, watch, reset } = useForm<FormData>({
    resolver: zodResolver(schema),
    defaultValues: loadDraft(),
  })
  
  // Persist on every change
  const values = watch()
  useEffect(() => {
    if (Object.keys(values).length > 0) {
      localStorage.setItem(STORAGE_KEY, JSON.stringify({
        values,
        savedAt: new Date().toISOString(),
      }))
    }
  }, [values])
  
  const onSubmit = async (data: FormData) => {
    const result = await submitApplication(data)
    if (result.ok) {
      // Clear draft on successful submit only
      localStorage.removeItem(STORAGE_KEY)
    }
  }
  
  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <DraftNotice />
      {/* ... */}
    </form>
  )
}

function loadDraft() {
  try {
    const saved = localStorage.getItem(STORAGE_KEY)
    if (!saved) return {}
    const { values, savedAt } = JSON.parse(saved)
    // Ignore drafts older than 7 days
    if (Date.now() - new Date(savedAt).getTime() > 7 * 24 * 60 * 60 * 1000) {
      localStorage.removeItem(STORAGE_KEY)
      return {}
    }
    return values
  } catch {
    return {}
  }
}
```

**UX rules:**

- When a draft is loaded, show a subtle "Restored from [X time ago]" notice with a "Start over" option
- Clear draft only on **successful** submission (not on failure — user may want to retry)
- Storage key is namespaced — `burlington:form:application`, `burlington:form:nda`, etc.
- Never persist sensitive data (passwords, card numbers, full SSN). Strip sensitive fields before saving.
- Drafts expire after 7 days automatically

### Network failure handling in forms

Long forms submitting over flaky connections need explicit handling.

```tsx
const onSubmit = async (data: FormData) => {
  try {
    const result = await submitApplication(data)
    if (!result.ok) {
      toast.error(result.error)
      return
    }
    localStorage.removeItem(STORAGE_KEY)
    router.push('/nda')
  } catch (error) {
    // Network error — don't lose the data
    if (error instanceof Error && error.message.includes('fetch')) {
      toast.error('Connection lost. Your answers are saved — please try again when you have signal.')
    } else {
      toast.error('Something went wrong. Your answers are saved — please try again.')
      Sentry.captureException(error)
    }
    // Form state remains; draft is still in localStorage
  }
}
```

### Error Handling

**Every async operation handles errors. No silent failures.**

```ts
// ✅ GOOD — explicit error handling, typed result
type Result<T> = { ok: true; data: T } | { ok: false; error: string }

export async function getBooking(id: string): Promise<Result<Booking>> {
  try {
    const { data, error } = await supabase
      .from('consultations')
      .select('*')
      .eq('id', id)
      .single()
    
    if (error) {
      console.error('[getBooking] Supabase error', error)
      return { ok: false, error: 'Failed to fetch booking' }
    }
    
    return { ok: true, data }
  } catch (error) {
    console.error('[getBooking] Unexpected error', error)
    return { ok: false, error: 'Unexpected error' }
  }
}

// ❌ BAD — errors swallowed, no context in logs
export async function getBooking(id: string) {
  const { data } = await supabase.from('consultations').select('*').eq('id', id).single()
  return data
}
```

**Log format:** `[functionName] message`, with error object as second arg. Easy to grep in Sentry.

**Never log sensitive data** (passwords, full card numbers, signed NDAs, full auth tokens). Log IDs, not content.

### Accessibility

**Every interactive element is accessible out of the box.**

- Semantic HTML first — `<button>`, `<a>`, `<nav>`, `<main>`, `<article>`, `<section>`
- Never use `<div onClick>` — it's not keyboard-accessible
- Every form input has an associated `<label>` (visually visible or `sr-only`)
- Every image has `alt` text (empty `alt=""` for decorative)
- Focus states visible — don't remove default outlines without replacement
- Modal dialogs: trap focus, return focus on close, `Esc` to close
- Color contrast: minimum 4.5:1 for body text, 3:1 for large text (verify with Tailwind's contrast utilities)
- `aria-label` when icon-only buttons (e.g., `<button aria-label="Close">×</button>`)
- `aria-invalid` and `aria-describedby` on form errors
- `role="status"` or `aria-live="polite"` for toast notifications

**Test with:** keyboard navigation (Tab through every interactive element), screen reader (VoiceOver on Mac, NVDA on Windows).

### Mobile-First

Every page works on mobile first, desktop second. Test on actual devices, not just browser devtools.

Breakpoints (Tailwind defaults):
- `sm:` 640px — larger phones, small tablets
- `md:` 768px — tablets
- `lg:` 1024px — small laptops
- `xl:` 1280px — desktops

**Touch targets:** minimum 44×44px (Apple) / 48×48dp (Material).

**Never rely on hover.** Every hover state needs a tap/focus equivalent.

├── tsconfig.json

├── railway.json                       # Railway config (start command, health check path)
└── .env.example

**Worker conventions:**

- **No user-facing endpoints.** The worker is not a REST API. Users never hit it. The only HTTP endpoint is `/health` for Railway's liveness check.
- **Stateless job handlers.** Each job function reads from Supabase, does its work, writes back. No in-memory state that survives restarts.
- **Idempotent by design.** Every job can safely run twice. Use database flags (`processed_at`, `sent_at`) to prevent duplicate work.
- **Log every job run.** Start, completion, errors. Sentry integration. Job runs are queryable after the fact.
- **Graceful shutdown.** SIGTERM handler that finishes in-flight work before exiting (Railway sends SIGTERM before restart).
- **Error budget.** If a single job throws, log and continue — don't crash the worker. A bad job for one client should not halt the queue for everyone.
- **Queue pattern:** Use a `pending_jobs` table in Supabase. Vercel inserts jobs (type, payload, run_after). Worker polls every 30 seconds, claims a batch, processes, marks complete or failed.

**Two triggers for work:**

```ts
// 1. Scheduled (cron) — runs at fixed times
cron.schedule('0 2 * * *', runSafelockInactivityCheck)      // Nightly 2am UTC
cron.schedule('0 * * * *', runConsultationReminders)        // Hourly
cron.schedule('0 9 * * 1', runWeeklyAdminDigest)            // Monday 9am UTC

// 2. Queued (on-demand) — Vercel enqueues, worker processes
// Vercel side:
await supabase.from('pending_jobs').insert({
  type: 'send_booking_confirmation',
  payload: { bookingId, clientEmail },
  run_after: new Date(),
})

// Worker side:
// Polls pending_jobs every 30s, finds the row, dispatches to handler
```

## Database Conventions

### Table design

- Primary keys: `id uuid primary key default gen_random_uuid()`
- Timestamps: `created_at timestamptz default now()`, `updated_at timestamptz default now()`
- Foreign keys: `references public.table_name` (with `on delete` clause specified)
- Status fields: `text` with check constraint, not Postgres enums (easier to evolve)
- Monetary amounts: `integer` (USD cents) — never `decimal` or `float`
- Timestamps: always `timestamptz` — never `timestamp`
- JSON: use `jsonb`, not `json`

### RLS policies

Every table has RLS enabled in the same migration that creates it:

```sql
-- Enable RLS
alter table public.consultations enable row level security;

-- Clients see their own bookings
create policy "Clients can view own consultations"
  on public.consultations for select
  to authenticated
  using (client_id = auth.uid());

-- Clients can create bookings for themselves
create policy "Clients can create own consultations"
  on public.consultations for insert
  to authenticated
  with check (client_id = auth.uid());

-- Advisers see bookings where they're the adviser
create policy "Advisers can view assigned consultations"
  on public.consultations for select
  to authenticated
  using (adviser_id = auth.uid());

-- Super admins see everything
create policy "Super admins have full access"
  on public.consultations for all
  to authenticated
  using (
    exists (
      select 1 from public.profiles
      where id = auth.uid() and role = 'super_admin'
    )
  );
```

**RLS test rule:** every policy has at least one passing and one failing test in `supabase/tests/`.

### Indexes

Add indexes where queries filter or join:
- Every foreign key column gets an index
- Columns used in `where` or `order by` frequently
- Composite indexes for common multi-column queries

Example:
```sql
create index idx_consultations_client_id on public.consultations(client_id);
create index idx_consultations_adviser_id on public.consultations(adviser_id);
create index idx_consultations_scheduled_at on public.consultations(scheduled_at);
```

### Triggers

For `updated_at` auto-updates:
```sql
create or replace function public.set_updated_at()
returns trigger as $$
begin
  new.updated_at = now();
  return new;
end;
$$ language plpgsql;

create trigger profiles_updated_at
  before update on public.profiles
  for each row execute function public.set_updated_at();
```

### Pagination — every list endpoint paginates

No endpoint returns an unbounded list. Even admin views. Even "internal" queries.

**Default page size 20, max 100.** If the UI genuinely needs more, paginate in the background.

**Offset-based** is fine for small/stable tables (profiles, team_profiles, engagements — < 100K rows, low churn).

**Cursor-based** is required for high-churn or large tables (payments, messages, audit_log, config_history, chat_messages, safelock_deposits).

```ts
// Cursor pattern for high-volume tables
export async function listPayments(opts: {
  limit?: number
  cursor?: string   // ISO timestamp of last seen row
}): Promise<{ data: Payment[]; nextCursor: string | null }> {
  const limit = Math.min(opts.limit ?? 20, 100)
  
  let query = supabase
    .from('payments')
    .select('*')
    .order('created_at', { ascending: false })
    .limit(limit)
  
  if (opts.cursor) query = query.lt('created_at', opts.cursor)
  
  const { data } = await query
  const nextCursor = data?.length === limit ? data[data.length - 1].created_at : null
  
  return { data: data ?? [], nextCursor }
}
```

Frontend shows "Load more" or infinite scroll based on `nextCursor`.

### Soft deletes — user-facing tables

Tables that hold user-facing records get a `deleted_at timestamptz` column. Records are never physically deleted through normal operations — deletions flip the timestamp.

**Tables that soft-delete:**
- `profiles` — account deletion preserves the record (GDPR requires ability to return data)
- `applications` — withdrawn or rejected stay for audit
- `consultations` — cancelled bookings preserve the history
- `documents` — deleted uploads stay until retention policy runs
- `messages` — deleted messages stay for moderation review
- `engagements` — terminated engagements preserve forever
- `team_profiles` — deactivated members become invisible but records remain

**Tables that do NOT soft-delete (append-only):**
- `audit_log`, `config_history`, `payments` — financial and audit records never change
- `processed_webhooks` — dedup records, can be purged on a schedule

**Rules for soft-deleted tables:**

```sql
-- All queries filter by default
create policy "Clients see own non-deleted applications"
  on public.applications for select
  to authenticated
  using (applicant_id = auth.uid() and deleted_at is null);

-- Index the filter column so it's fast
create index idx_applications_active on public.applications(applicant_id) 
  where deleted_at is null;
```

Supabase client helpers should default to filtering `deleted_at IS NULL`:

```ts
// lib/db/applications.ts
export function getApplications(clientId: string, includeDeleted = false) {
  let query = supabase.from('applications').select('*').eq('applicant_id', clientId)
  if (!includeDeleted) query = query.is('deleted_at', null)
  return query
}
```

**Hard delete only happens via a scheduled retention job** in the Railway worker — e.g., "hard-delete all messages marked `deleted_at` more than 2 years ago." This is the right place for GDPR-compliant purge logic.

### Migration pattern — expand then contract, never both

The number one way production migrations break: dropping a column in the same deploy that changes the code using it. Old code still running sees a missing column; new code hasn't deployed yet. Prod goes down.

**Rule: every destructive migration is two deploys apart.**

```
Sprint 5, Week 1:
  Migration A: Add new column `client_status` alongside existing `status`
  Code deploy: write to BOTH columns; read from the new one

Sprint 6, Week 3:
  (verify over time that nothing reads the old column)
  Migration B: Drop the old `status` column
  Code deploy: write to only the new column
```

**Expand phase** — additive, reversible:
- Add new column
- Add new table
- Add new index
- Create a trigger that keeps old and new in sync

**Contract phase** — destructive, comes later:
- Drop old column
- Drop old table
- Drop obsolete index
- Remove sync trigger

**Never in the same deploy:** schema drops + code that depends on the drop. The gap between expand and contract should be at least one production deploy, ideally a week.

**Migrations are forward-only in production.** We don't run `down()` migrations on production databases — we write a new migration that reverses the change. This preserves history and avoids "which direction did this run?" confusion.

### Backup strategy

Supabase Pro includes automated daily backups with 7-day point-in-time recovery. This is the baseline. For additional safety:

- **Sprint 12 deliverable:** verify restore process by doing a test restore to a staging environment. Document the runbook.
- **Before every destructive migration in production:** trigger an on-demand backup via Supabase dashboard. Document this in the PR.
- **Quarterly:** Chris receives a "last quarter's backup verification" report — confirmed tests of restore from the most recent backup.

## Money & Currency

This is a payments app. Money rules are non-negotiable.

### Storage

- **All monetary values are stored as integer minor units.** USD in cents, NGN in kobo.
  - `$20,000.00` → `2_000_000` (cents)
  - `₦32,500,000.00` → `3_250_000_000` (kobo)
- Column type is always `integer` (or `bigint` if you're genuinely expecting values over ~$21M — rare for us).
- Column name should end in `_amount` with the currency implied by context or an adjacent `currency` column: `amount_usd integer`, `amount_local integer`, `currency text`.
- Never `decimal`, never `numeric(10,2)`, never `float`, never `money` (Postgres `money` type is specifically banned — it's locale-dependent and fragile).

### Display

Conversion from cents to displayable strings happens **only at the display layer.** Business logic works in cents.

```ts
// packages/shared/utils/format-currency.ts — the ONE place currency formatting lives
export function formatCurrency(amountMinorUnits: number, currency: 'USD' | 'NGN'): string {
  const formatter = new Intl.NumberFormat(currency === 'USD' ? 'en-US' : 'en-NG', {
    style: 'currency',
    currency,
    minimumFractionDigits: 2,
    maximumFractionDigits: 2,
  })
  return formatter.format(amountMinorUnits / 100)
}

// Usage:
formatCurrency(2_000_000, 'USD')    // "$20,000.00"
formatCurrency(3_250_000_000, 'NGN') // "₦32,500,000.00"
```

**Display conventions:**
- First reference in a UI context: show symbol + ISO code (`$20,000 USD`, `₦32,500,000 NGN`)
- Subsequent references in the same context: symbol alone is fine (`$20,000`)
- Never mix currencies in the same calculation display without explicit conversion shown

### Currency conversion

- Exchange rates come from one provider (pick at launch — suggest `exchangerate.host` or `Open Exchange Rates` free tier)
- Rates cached in a `exchange_rates` table, refreshed every 6 hours by the Railway worker
- Never convert currencies inline in business logic — always go through `lib/services/currency.ts`
- When displaying both currencies (e.g., Nigerian client seeing USD equivalent), show the exchange rate and refresh timestamp

### Payments & refunds

- **Every payment stores the amount in its original currency** (not just a USD equivalent)
- **Refunds refund exactly what was charged.** Pull the original `amount_local` and `currency` from the `payments` row. Never recompute from an exchange rate.
- Provider fees (Stripe 2.9%, Paystack 1.5%, NOWPayments 1%) are tracked in a separate `provider_fee` column on `payments` for accurate net-revenue reporting

### Stablecoins

- USDC and USDT are treated as USD equivalents for accounting purposes (pegged 1:1)
- Store the actual token received (`currency: 'USDC' | 'USDT'`) alongside the USD-equivalent cents amount
- Auto-conversion via NOWPayments settles in USD; the transaction shows both the crypto received and the USD settled

## Dates, Times & Timezones

Bookings live and die on timezone correctness. Get this wrong and clients miss consultations.

### Storage

- **Every timestamp is `timestamptz` and stored in UTC.** No exceptions.
- Column naming: `_at` suffix for instants (`created_at`, `scheduled_at`, `signed_at`), `_on` suffix for dates without time (`birthday_on`, `due_on`)
- Never store timezone names in the same column as the timestamp (separate column: `timezone text` if needed)

### Clock service

Never call `new Date()` directly in business logic. Use a clock service so tests can mock time.

```ts
// packages/shared/utils/clock.ts
export const clock = {
  now: () => new Date(),
  today: () => new Date(new Date().setHours(0, 0, 0, 0)),
}

// In business logic:
import { clock } from '@/lib/clock'
const now = clock.now()

// In tests:
vi.spyOn(clock, 'now').mockReturnValue(new Date('2026-04-15T10:00:00Z'))
```

UI display code (e.g., `new Date(booking.scheduled_at).toLocaleString()`) is fine to call `new Date()` directly — it's just formatting.

### Date library

**Use `date-fns`.** One library across the entire codebase.

- `date-fns` for all date manipulation, formatting, parsing
- `date-fns-tz` for timezone conversions
- Do not use `moment` (deprecated), `dayjs`, `luxon`, or native `Date` math (`Date.now() + 86400000` is a bug waiting to happen)
- When Temporal is stable across all target browsers, we'll migrate. Not yet.

### Display

- **Display times in the user's local timezone.** Detect from the browser (`Intl.DateTimeFormat().resolvedOptions().timeZone`)
- **Store the user's preferred timezone** in `profiles.timezone` for server-rendered emails
- Server-rendered emails render times in the recipient's timezone (pulled from their profile)
- Always show the timezone abbreviation alongside the time (`10:00 AM EST`, `3:00 PM WAT`) — never ambiguous

### Booking slot calculation

- Adviser availability is stored in the adviser's timezone (stored on their profile)
- Slot calculations done in the adviser's timezone, then converted to the client's timezone for display
- The `consultations.scheduled_at` is always UTC — both sides convert from their local zone before storage
- Unique constraint on `(adviser_id, scheduled_at)` for `status IN ('confirmed', 'pending_payment')` — see Booking Race Conditions below

### String handling

- Never concatenate strings to build dates (`'2026-04-' + day` is banned)
- Parse ISO 8601 strings with `parseISO` from `date-fns`, never `new Date(str)` (inconsistent across engines)
- When outputting dates for APIs, use `toISOString()` or `formatISO` from `date-fns`

## Names, Addresses & Phone Numbers

Assumptions about names, addresses, and phone numbers are how software quietly discriminates against users outside the developer's culture. Burlington's clients are global; the platform respects that.

### Names

**Primary field:** `full_name text not null`. One field. The user controls how their name appears.

**Secondary fields** (optional, collected only when needed for specific deliverables like NDAs or engagement letters): `given_name`, `family_name`. Some legal documents require a breakdown; ask for it at that moment, not at signup.

**Never:**
- Force "First name" + "Last name" as two required fields at signup
- Assume family name comes last (not true in China, Hungary, Japan, Korea, Vietnam)
- Assume a single family name (many Spanish-speaking cultures use two)
- Limit name length below 100 characters (some valid names exceed Western assumptions)
- Strip Unicode characters (names contain accents, diacritics, non-Latin scripts)
- Validate names against a regex beyond "not empty and under max length"

**Display rules:**
- Greetings in email: use the user's stated preferred name or the full name as they entered it
- Respect professional titles when the user provides them (`Dr.`, `Prof.`, etc.) but don't assume
- Never truncate a name in a way that changes its meaning (`Okonkwo-Chukwuebuka` truncated to `Okonkwo...` is fine for table views; avoid anywhere the full name matters)

### Addresses

**Use a flexible schema.** U.S.-specific fields (`state`, `zip`) hardcoded into the schema are a common bug that makes the form unusable for 90% of the world.

```sql
create table public.addresses (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid references public.profiles,
  address_line_1 text not null,
  address_line_2 text,
  city text not null,
  administrative_area text,        -- state / province / county / region / LGA
  postal_code text,                -- not required everywhere (Ireland had no postcodes until recently)
  country_iso text not null,       -- ISO 3166-1 alpha-2 ("NG", "US", "GB")
  created_at timestamptz default now()
);
```

**Form behaviour:**
- Country field determines which other fields are required (US requires state and zip; many countries don't require postal code)
- Country-specific labels (US: "State", "ZIP code" / UK: "County", "Postcode" / Nigeria: "State" / Germany: "Land", "Postleitzahl" — all with English fallbacks)
- Keep validation lenient — international postal codes have wildly different formats (UK: `SW1A 1AA`, Canada: `K1A 0B1`, Netherlands: `1012 JS`, Brazil: `01310-100`)

**Preferred approach for MVP:** Use Google Places Autocomplete for address entry. Handles international formats, parses the response into our schema. `@react-google-maps/api` or the newer `@googlemaps/extended-component-library`. Post-MVP, consider falling back to free-form for users who can't find their address.

### Phone numbers

**Use `libphonenumber-js`.** Industry-standard library, ~100KB gzipped (acceptable for this use case), handles every country's format correctly.

**Storage:** Always E.164 format: `+[country code][national number]` with no spaces, dashes, or parentheses. Examples: `+2348032966493`, `+14155552671`, `+447911123456`.

```ts
// lib/utils/phone.ts
import { parsePhoneNumberFromString, CountryCode } from 'libphonenumber-js'

export function normalizePhone(input: string, defaultCountry?: CountryCode): string | null {
  const parsed = parsePhoneNumberFromString(input, defaultCountry)
  if (!parsed || !parsed.isValid()) return null
  return parsed.format('E.164')
}

export function displayPhone(e164: string): string {
  const parsed = parsePhoneNumberFromString(e164)
  return parsed?.formatInternational() ?? e164
}
```

**Form UI:**
- Country code picker (dropdown with flags is fine, but list must be comprehensive and searchable)
- Detect user's country from IP as a default, never as an enforcement
- Accept national format and normalize to E.164 on save
- Display in the user's preferred format (international by default)

**Never:**
- Store phone numbers with hyphens, parentheses, or formatting (`(234) 803-296-6493` is banned)
- Strip the `+` — E.164 requires it
- Validate phone length with a fixed regex (different countries have different lengths)
- Default the country code picker to `+234` or `+1` — detect, don't assume

## Status Fields & State Machines

Every entity with a `status` field has a legal set of transitions. Document them. Enforce them in code. Don't let status change arbitrarily.

### Applications

```
pending → under_review → (approved | rejected | waitlisted | needs_info)
needs_info → under_review → (approved | rejected | waitlisted)
waitlisted → approved  (manual promotion by super admin)
```

- `pending → under_review`: automatic when application lands in admin queue
- `under_review → approved/rejected/waitlisted`: requires `admin` or `super_admin` role, logged in audit
- `under_review → needs_info`: requires specifying what's missing; triggers email to applicant
- Rejection is terminal but reapplication is allowed (creates a new application record, not an update)

### Consultations (Bookings)

```
pending_payment → confirmed → (completed | cancelled_by_client | cancelled_by_burlington | no_show)
pending_payment → expired  (if not paid within 15 minutes)
confirmed → rescheduled (creates a new booking, marks old one cancelled)
```

- `pending_payment → confirmed`: triggered by payment webhook, never by client action
- `confirmed → completed`: adviser marks complete post-session, 24 hours after `scheduled_at`
- `confirmed → cancelled_by_client`: allowed up to 24 hours before `scheduled_at`
- `confirmed → cancelled_by_burlington`: anytime, with reason, triggers refund

### Payments

```
pending → (succeeded | failed)
succeeded → refunded  (admin-triggered only, via webhook from provider)
succeeded → partially_refunded
```

- Terminal states: `failed`, `refunded`
- Refund is initiated through the provider, not by direct DB update — we wait for the provider webhook to mark as refunded

### Safelocks

```
active → (completed | forfeited | refunded)
active → forfeiture_pending → (forfeited | reactivated)
```

- `active → completed`: total deposits ≥ target amount; triggers engagement letter flow
- `active → forfeiture_pending`: 11 months without deposit; 30-day notice email sent
- `forfeiture_pending → reactivated`: any deposit within the 30-day notice window
- `forfeiture_pending → forfeited`: 30 days elapse with no deposit; funds forfeited per terms
- `active → refunded`: client-initiated refund per refund policy (5% or $100, whichever is less)

### Engagements

```
draft → sent → signed → active → (completed | terminated)
```

- `draft → sent`: admin sends letter for signing
- `sent → signed`: client signs electronically
- `signed → active`: first milestone payment received
- `active → completed`: all milestones paid and work delivered
- `active → terminated`: early termination per engagement terms (manual, with reason)

### Enforcement

- All transitions happen through a single function per entity: `transitionApplication()`, `transitionBooking()`, etc.
- These functions check the current status and validate the new status is reachable
- Invalid transitions throw — never silently succeed

```ts
// lib/services/transitions.ts
const bookingTransitions: Record<BookingStatus, BookingStatus[]> = {
  pending_payment: ['confirmed', 'expired'],
  confirmed: ['completed', 'cancelled_by_client', 'cancelled_by_burlington', 'no_show'],
  completed: [],
  cancelled_by_client: [],
  cancelled_by_burlington: [],
  no_show: [],
  expired: [],
}

export function canTransitionBooking(from: BookingStatus, to: BookingStatus): boolean {
  return bookingTransitions[from]?.includes(to) ?? false
}
```

## Webhook Idempotency

Stripe, Paystack, and NOWPayments all retry webhooks. Without idempotency, we'll double-charge, double-email, or double-activate engagements.

### The rule

**Every webhook handler is idempotent.** Running it twice with the same event produces the same result as running it once.

### Implementation

```sql
create table public.processed_webhooks (
  id uuid primary key default gen_random_uuid(),
  provider text not null,              -- 'stripe' | 'paystack' | 'nowpayments'
  event_id text not null,              -- Provider's event ID
  event_type text not null,
  processed_at timestamptz default now(),
  result text not null,                -- 'success' | 'error'
  error_message text,
  unique (provider, event_id)          -- Prevents double-processing
);

create index idx_processed_webhooks_lookup on public.processed_webhooks(provider, event_id);
```

### Handler pattern

```ts
// app/api/webhooks/stripe/route.ts
export async function POST(req: NextRequest) {
  // 1. Verify signature FIRST (before any processing)
  const signature = req.headers.get('stripe-signature')
  if (!signature) return Response.json({ error: 'Missing signature' }, { status: 400 })
  
  const body = await req.text()
  let event
  try {
    event = stripe.webhooks.constructEvent(body, signature, process.env.STRIPE_WEBHOOK_SECRET!)
  } catch {
    return Response.json({ error: 'Invalid signature' }, { status: 400 })
  }
  
  // 2. Check if we've seen this event before
  const { data: existing } = await supabaseAdmin
    .from('processed_webhooks')
    .select('id, result')
    .eq('provider', 'stripe')
    .eq('event_id', event.id)
    .maybeSingle()
  
  if (existing) {
    // Already processed — return success and skip
    console.info('[stripe-webhook] Skipping duplicate event', { eventId: event.id })
    return Response.json({ received: true, duplicate: true })
  }
  
  // 3. Process the event
  try {
    await handleStripeEvent(event)
    
    // 4. Mark as processed
    await supabaseAdmin.from('processed_webhooks').insert({
      provider: 'stripe',
      event_id: event.id,
      event_type: event.type,
      result: 'success',
    })
    
    return Response.json({ received: true })
  } catch (error) {
    // Log the error and mark as errored (still recorded, so retries don't duplicate work)
    console.error('[stripe-webhook] Handler failed', error, { eventId: event.id })
    
    await supabaseAdmin.from('processed_webhooks').insert({
      provider: 'stripe',
      event_id: event.id,
      event_type: event.type,
      result: 'error',
      error_message: error instanceof Error ? error.message : String(error),
    })
    
    // Return 500 so the provider retries (if the error is transient)
    return Response.json({ error: 'Handler failed' }, { status: 500 })
  }
}
```

### Race conditions

For events that might arrive simultaneously (rare but possible — e.g., Stripe sending `payment_intent.succeeded` twice in ~1 second), wrap the handler body in a Postgres advisory lock:

```sql
-- At the start of processing
select pg_advisory_xact_lock(hashtext('stripe_event_' || $1));
```

This blocks concurrent processing of the same event ID within a transaction. First transaction wins, second waits, sees the row now exists, returns duplicate.

| NDA (generated) | N/A | `application/pdf` (system-generated only, no user upload) |
| Engagement letter (generated) | N/A | `application/pdf` (system-generated only) |

### Server-side validation

Never trust the client's declared MIME type or filename extension. Validate on the server:

```ts
// Read first 4 kilobytes and inspect magic bytes
import { fileTypeFromBuffer } from 'file-type'

const detected = await fileTypeFromBuffer(buffer)
if (!detected || !ALLOWED_MIMES.includes(detected.mime)) {
  throw new Error('Unsupported file type')
}
```

Reject executables outright — never accept `application/x-msdownload`, `application/x-executable`, etc.

### Naming

**Never use a user-supplied filename as the storage path.** User filenames can contain path traversal (`../../etc/passwd`), unicode tricks, or cause collisions.

Storage path format: `{userId}/{timestamp}-{slug}.{ext}`

```ts
const storagePath = `${userId}/${Date.now()}-${slugify(originalName)}.${extension}`
// e.g., "a1b2c3/1713198234567-john-smith-cv.pdf"
```

Store the original filename in a separate column (`documents.original_filename`) for display to the user.

### Buckets

| Bucket | Public access | Use |
|---|---|---|
| `avatars` | Public (signed URLs not required) | User profile photos |
| `documents` | Private (signed URLs only) | Client CVs, certs, evidence |
| `ndas` | Private (signed URLs only) | Signed NDAs |
| `engagement-letters` | Private (signed URLs only) | Signed engagement letters |

RLS policies per bucket match the access rules for the related table (e.g., `documents` bucket access mirrors `documents` table RLS).

### Signed URLs

- Private files are always served via time-limited signed URLs
- TTL: 1 hour for download links shown to the logged-in user, 24 hours for lawyer-access links (regenerated on each view)
- Never store signed URLs in the database — always generate on demand

### Virus scanning

**MVP:** Reject dangerous MIME types (executables), enforce size limits, rely on Supabase's built-in storage safeguards.

**v2 (post-launch):** ClamAV scanning via a Supabase Edge Function or a dedicated scanning step in the Railway worker before a file is marked `status = 'available'`. Files in uploaded state are not visible to admins or clients until scanned.

## Booking Race Conditions

Two clients trying to book the same slot is an inevitable bug without explicit protection.

### Database-level constraint

```sql
-- Prevent two confirmed/pending bookings for the same adviser at the same time
create unique index idx_consultations_adviser_slot
  on public.consultations(adviser_id, scheduled_at)
  where status in ('pending_payment', 'confirmed');
```

This constraint is the source of truth. Application logic can be wrong; the DB cannot.

### Insert pattern

```ts
// lib/db/bookings.ts
export async function createBooking(input: CreateBookingInput): Promise<Result<Booking>> {
  const { data, error } = await supabase
    .from('consultations')
    .insert({
      client_id: input.clientId,
      adviser_id: input.adviserId,
      scheduled_at: input.scheduledAt,
      tier: input.tier,
      status: 'pending_payment',
    })
    .select()
    .single()
  
  if (error) {
    // Unique violation = slot taken
    if (error.code === '23505') {
      return { ok: false, error: 'This slot was just booked by someone else. Please pick another.' }
    }
    console.error('[createBooking]', error)
    return { ok: false, error: 'Could not create booking' }
  }
  
  return { ok: true, data }
}
```

### Expiring pending bookings

`pending_payment` bookings hold the slot for 15 minutes. If payment doesn't complete:

- Railway worker cron job runs every 5 minutes
- Finds bookings in `pending_payment` status older than 15 minutes
- Transitions them to `expired`
- Slot is now free again

### UI consideration

Optimistic UI (client picks slot, immediately shows as reserved) is fine, but the source of truth is the database. Always re-fetch availability after any user action, and show a clear error if the server rejects the insert.

## Privacy & Compliance

Burlington's platform processes personal and sensitive data (professional records, immigration goals, identity documents, payment information) for clients across many jurisdictions. Privacy compliance is multi-jurisdictional. The simplest, most defensible path is to meet the strictest applicable standard — GDPR — everywhere.

### Regulations that apply

| Regulation | Scope | Trigger |
|---|---|---|
| **GDPR** | EU residents | Any EU resident signs up or visits |
| **UK GDPR** | UK residents | Any UK resident signs up or visits |
| **CCPA/CPRA** | California residents | Any California resident signs up or visits |
| **NDPR** | Nigerian residents | Any Nigerian resident signs up or visits (modeled on GDPR) |
| **LGPD** | Brazilian residents | Any Brazilian resident signs up or visits |
| **PIPEDA** | Canadian residents | Any Canadian resident signs up or visits |

We don't check residency before applying protections. We apply GDPR-level protections to everyone. This is both ethically correct and operationally simpler.

### What GDPR-equivalent compliance requires

**Lawful basis for processing.** Document the lawful basis for each data category. For Burlington: consent (marketing emails), contract performance (service delivery), and legitimate interest (fraud prevention, admin operations). Keep this documented in `PRIVACY.md` at the repo root.

**Consent for non-essential cookies and tracking.** The Google Analytics pixel, Meta Pixel, and Microsoft Clarity all require consent for EU/UK/California users. Use GA4 Consent Mode or a consent management platform (suggestion: Cookiebot or Iubenda — both have free tiers).

**Data subject rights.** Users can request:
- **Access** — export of all data we hold about them (build: `/settings/data/export` → downloads JSON)
- **Rectification** — ability to correct their data (build: editable profile + `/settings/data/correction-request`)
- **Erasure ("right to be forgotten")** — deletion of their account and all associated data (build: `/settings/data/delete-account` with 30-day grace period, confirmation email, then hard delete)
- **Portability** — receive their data in a machine-readable format (JSON export covers this)
- **Objection** — opt out of marketing emails (unsubscribe link on every marketing email)

Requests must be actioned within 30 days. Track them in an `data_requests` table:

```sql
create table public.data_requests (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid references public.profiles not null,
  type text not null,              -- 'access' | 'rectification' | 'erasure' | 'portability' | 'objection'
  status text default 'pending',   -- 'pending' | 'in_progress' | 'completed' | 'rejected'
  requested_at timestamptz default now(),
  completed_at timestamptz,
  notes text
);
```

**Data retention policy.** Documented maximum retention periods per data category. Suggestion for Burlington:
- Client account + case files: 7 years after engagement ends (common legal retention standard)
- Unapproved applications: 12 months, then auto-purge
- Marketing emails (unsubscribed): 30 days, then delete
- Chat logs: 2 years, then anonymise

**Breach notification.** If a breach occurs, notify affected users within 72 hours (GDPR standard). Keep a `security_incidents` log. This is a documented process, not code.

**Privacy policy.** `/privacy` page must exist and be up-to-date. Updates trigger a banner asking users to re-read.

**Data Processing Agreements (DPAs).** Sign DPAs with every subprocessor: Supabase, Vercel, Stripe, Paystack, NOWPayments, Resend, Sentry, and the chatbot service. All are standard — linked from each provider's legal page.

### Cookie consent

**Required in:** EU, UK, California (somewhat), Brazil.
**Practical approach:** Show the cookie banner everywhere. Default all non-essential cookies to OFF (GDPR standard). Only load analytics, Meta Pixel, Clarity, etc. after the user consents.

Implementation pattern:
```ts
// app/layout.tsx — scripts load conditionally based on consent
{consent.analytics && <Script src="https://www.googletagmanager.com/gtag/js?id=G-WGX2C54Y62" />}
{consent.marketing && <MetaPixel />}
```

Store consent in a cookie (`consent_preferences`) with 12-month expiry. Re-prompt annually.

### What NOT to do (privacy red flags)

- Never share client data with third parties without DPAs and explicit user consent
- Never email marketing content to a user who hasn't opted in (GDPR: opt-in; CCPA: opt-out — we use opt-in everywhere)
- Never retain data longer than our stated retention policy
- Never expose user data in URL parameters (`/profile?email=user@example.com` is banned)
- Never log full personal data to Sentry or Vercel logs (use opaque IDs)
- Never hardcode test data with real email addresses (use `*@example.com` patterns)
- Never collect data we don't need ("data minimisation" is a GDPR principle)

## Security Headers & CSP

Headers are the first line of defence before any of our code runs. A correctly configured Content Security Policy prevents XSS, clickjacking, and data exfiltration even if an attacker gets a script onto the page.

### Required headers on every response

Every response from the portal, admin, and marketing site must include these headers. Configured in `next.config.ts` using the `headers()` function so they apply globally.

```ts
// next.config.ts
const securityHeaders = [
  {
    key: 'Content-Security-Policy',
    value: cspHeader,  // See below
  },
  {
    key: 'X-Frame-Options',
    value: 'DENY',  // Prevent clickjacking — no one frames us
  },
  {
    key: 'X-Content-Type-Options',
    value: 'nosniff',  // Prevent MIME sniffing attacks
  },
  {
    key: 'Referrer-Policy',
    value: 'strict-origin-when-cross-origin',  // Don't leak referrer data
  },
  {
    key: 'Permissions-Policy',
    value: 'camera=(), microphone=(), geolocation=(), interest-cohort=()',  // No FLoC, no unused browser APIs
  },
  {
    key: 'Strict-Transport-Security',
    value: 'max-age=63072000; includeSubDomains; preload',  // HSTS 2 years
  },
]

module.exports = {
  async headers() {
    return [
      {
        source: '/:path*',
        headers: securityHeaders,
      },
    ]
  },
}
```

### Content Security Policy

CSP is the most powerful and most fragile header. It must be tuned to our actual runtime dependencies. The policy below is the starting point; tighten over time using report-only mode to find violations.

```ts
const cspHeader = `
  default-src 'self';
  script-src 'self' 'unsafe-inline' 'unsafe-eval' 
    https://js.stripe.com 
    https://checkout.paystack.com
    https://js.paystack.co
    https://www.googletagmanager.com
    https://www.google-analytics.com;
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  img-src 'self' data: blob: 
    https://*.supabase.co
    https://www.google-analytics.com;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' 
    https://*.supabase.co
    wss://*.supabase.co
    https://api.stripe.com
    https://api.paystack.co
    https://api.nowpayments.io
    ${process.env.CHATBOT_API_URL}
    https://www.google-analytics.com;
  frame-src 'self' 
    https://js.stripe.com
    https://checkout.paystack.com;
  object-src 'none';
  base-uri 'self';
  form-action 'self';
  frame-ancestors 'none';
  upgrade-insecure-requests;
`.replace(/\s{2,}/g, ' ').trim()
```

### Deployment strategy

1. **Start in report-only mode** — `Content-Security-Policy-Report-Only` first, so violations are logged but nothing breaks
2. **Monitor for 1 week** via Sentry integration — capture CSP violations as warnings
3. **Fix legitimate violations** by adding trusted sources to the policy
4. **Flip to enforce mode** — change header key to `Content-Security-Policy`
5. **Monitor forever** — every new third-party integration requires a CSP update

### The `'unsafe-inline'` escape hatches

The policy above uses `'unsafe-inline'` for scripts and styles. This is a weakness. Next.js inlines some bootstrapping code by default, and Tailwind injects inline styles. Removing `'unsafe-inline'` requires:

- Using nonces or hashes for every inline script (complex, Next.js has middleware for this)
- Switching to external stylesheets only (loses some Tailwind performance wins)

**Decision:** Accept `'unsafe-inline'` at launch. Plan a Sprint 12 hardening pass to implement nonce-based CSP — this is a premium detail worth doing but not launch-blocking.

### Testing

- Use Mozilla Observatory (observatory.mozilla.org) to grade headers before launch. Target: A+
- Use securityheaders.com for a second opinion
- Lighthouse security audit as part of CI

## Rate Limiting

Without rate limits, a single script can exhaust our budget or bring the service down. Every public-facing endpoint that touches a database or external API must be rate-limited.

### Implementation

Use **Upstash Ratelimit** with Redis — free tier is generous, integrates directly with Vercel Edge middleware.

```ts
// lib/services/ratelimit.ts
import { Ratelimit } from '@upstash/ratelimit'
import { Redis } from '@upstash/redis'

const redis = Redis.fromEnv()

export const rateLimiters = {
  // Auth endpoints — strict
  auth: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(5, '15 m'),  // 5 attempts per 15 min
    analytics: true,
    prefix: 'ratelimit:auth',
  }),

  // Application form — strict (prevent spam applications)
  application: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(3, '1 h'),
    analytics: true,
    prefix: 'ratelimit:application',
  }),

  // Chatbot proxy — moderate
  chat: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(30, '1 m'),  // 30 messages per minute per IP
    analytics: true,
    prefix: 'ratelimit:chat',
  }),

  // Assessment submissions — lenient (it's a lead gen tool)
  assessment: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(10, '1 h'),
    analytics: true,
    prefix: 'ratelimit:assessment',
  }),

  // General API — default
  api: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(60, '1 m'),
    analytics: true,
    prefix: 'ratelimit:api',
  }),
}

export async function checkRateLimit(
  limiter: keyof typeof rateLimiters,
  identifier: string
) {
  const result = await rateLimiters[limiter].limit(identifier)
  return {
    allowed: result.success,
    limit: result.limit,
    remaining: result.remaining,
    reset: result.reset,
  }
}
```

### Usage pattern

```ts
// In an API route or Server Action
import { checkRateLimit } from '@/lib/services/ratelimit'
import { headers } from 'next/headers'

export async function POST(req: NextRequest) {
  const ip = headers().get('x-forwarded-for') ?? 'unknown'
  const { allowed, remaining, reset } = await checkRateLimit('application', ip)
  
  if (!allowed) {
    return Response.json(
      { error: 'Too many requests. Please try again later.' },
      {
        status: 429,
        headers: {
          'X-RateLimit-Limit': '3',
          'X-RateLimit-Remaining': String(remaining),
          'X-RateLimit-Reset': String(reset),
          'Retry-After': String(Math.ceil((reset - Date.now()) / 1000)),
        },
      }
    )
  }
  
  // ... continue with request
}
```

### Identifier strategy

- **Public endpoints (no auth):** rate limit by IP address (`x-forwarded-for` header from Vercel)
- **Authenticated endpoints:** rate limit by user ID (more accurate, prevents shared-IP false positives)
- **Webhooks:** rate limit per provider (e.g., Stripe IPs are known, allowlist them and skip rate limits)
- **Admin endpoints:** higher limits for super admins, standard limits for regular admins

### Which routes need rate limits

| Route | Limiter | Reason |
|---|---|---|
| `/login`, `/signup`, `/forgot-password` | `auth` | Credential stuffing, brute force |
| `/apply` (application submission) | `application` | Spam applications |
| `/api/nda/sign` | `application` | Abuse of signing flow |
| `/api/chat` (chatbot proxy) | `chat` | Token cost control |
| `/assess` (free assessment) | `assessment` | Bot traffic on lead gen |
| `/api/checkout` | `api` | Fraudulent checkout attempts |
| Password reset, email verification | `auth` | Email spam |
| Any public GET on client data | `api` | Scraping |

### Monitoring

- Upstash analytics dashboard shows hit rates per limiter
- Sentry captures every 429 response with context
- Chris sees rate limit trips in the admin dashboard (especially spikes which may indicate attacks)

## External API Resilience

Stripe, Paystack, NOWPayments, Resend, the chatbot service — all will fail occasionally. Transient failures must not become user-visible errors. Persistent failures must not cascade.

### The rules

Every call to an external service must have:

1. **An explicit timeout** — no call waits forever
2. **Retry logic for idempotent operations** — with exponential backoff and jitter
3. **A maximum retry count** — don't retry infinitely
4. **Proper error classification** — distinguish transient vs permanent failures
5. **Circuit breaker for persistent failures** — stop hammering a failing service

### Timeouts

Every `fetch()` call and every SDK call specifies a timeout. No exceptions.

```ts
// lib/utils/fetch-with-timeout.ts
export async function fetchWithTimeout(
  url: string,
  options: RequestInit = {},
  timeoutMs = 10_000  // 10s default
): Promise<Response> {
  const controller = new AbortController()
  const timeout = setTimeout(() => controller.abort(), timeoutMs)
  
  try {
    const response = await fetch(url, {
      ...options,
      signal: controller.signal,
    })
    return response
  } finally {
    clearTimeout(timeout)
  }
}
```

**Timeout defaults by operation:**

| Operation | Connect timeout | Read timeout | Why |
|---|---|---|---|
| Payment provider API (charge, checkout session) | 3s | 30s | Can't interrupt a charge confirmation |
| Payment webhook handler (our side) | 2s | 25s | Provider retries if we're slow |
| Email send | 2s | 10s | User doesn't wait — async anyway |
| Chatbot API | 2s | 20s | User is waiting, but LLMs need time |
| Exchange rate fetch | 1s | 5s | Fail fast, use cached rate |
| Sanctions/country lookup | 1s | 3s | Local data, should be instant |
| File upload to storage | 5s | 120s | Large uploads allowed |
| Health checks | 500ms | 2s | Must be fast or we flag as unhealthy |

### Retry with exponential backoff

```ts
// lib/utils/retry.ts
interface RetryOptions {
  maxAttempts?: number
  initialDelayMs?: number
  maxDelayMs?: number
  factor?: number
  jitterMs?: number
  isRetryable?: (error: unknown) => boolean
}

export async function retry<T>(
  operation: () => Promise<T>,
  options: RetryOptions = {}
): Promise<T> {
  const {
    maxAttempts = 3,
    initialDelayMs = 500,
    maxDelayMs = 10_000,
    factor = 2,
    jitterMs = 200,
    isRetryable = defaultIsRetryable,
  } = options
  
  let lastError: unknown
  
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await operation()
    } catch (error) {
      lastError = error
      
      if (attempt === maxAttempts) break
      if (!isRetryable(error)) break
      
      // Exponential backoff with jitter
      const baseDelay = Math.min(
        initialDelayMs * Math.pow(factor, attempt - 1),
        maxDelayMs
      )
      const jitter = Math.random() * jitterMs
      const delay = baseDelay + jitter
      
      console.warn(`[retry] Attempt ${attempt} failed, retrying in ${delay}ms`, error)
      await new Promise(resolve => setTimeout(resolve, delay))
    }
  }
  
  throw lastError
}

function defaultIsRetryable(error: unknown): boolean {
  // Retry on network errors and 5xx responses
  if (error instanceof TypeError) return true  // Network error
  if (error instanceof Error && error.name === 'AbortError') return true  // Timeout
  
  // Response-based checks
  if (error && typeof error === 'object' && 'status' in error) {
    const status = (error as { status: number }).status
    return status >= 500 || status === 429  // 5xx or rate limited
  }
  
  return false
}
```

### Retryable vs non-retryable operations

**Retry safely:**
- GET requests (always idempotent)
- POST with an idempotency key (Stripe, Paystack provide this)
- PUT requests (idempotent by definition)
- DELETE (idempotent — same target is gone either way)
- Webhook signature verification failures (maybe corrupted in transit)

**Never retry automatically:**
- POST without an idempotency key (could double-charge)
- Payment confirmation that doesn't use idempotency keys
- Anything where "did it happen?" is ambiguous

### Circuit breaker pattern

For services that fail persistently (rare but catastrophic — e.g., Paystack is down for 15 minutes), a circuit breaker stops hammering them and fails fast.

```ts
// lib/utils/circuit-breaker.ts
export class CircuitBreaker {
  private failures = 0
  private lastFailureTime = 0
  private state: 'closed' | 'open' | 'half-open' = 'closed'
  
  constructor(
    private readonly threshold = 5,          // Failures before opening
    private readonly timeoutMs = 60_000,     // Time to wait before half-open
    private readonly name = 'default'
  ) {}
  
  async execute<T>(operation: () => Promise<T>): Promise<T> {
    if (this.state === 'open') {
      if (Date.now() - this.lastFailureTime > this.timeoutMs) {
        this.state = 'half-open'
      } else {
        throw new Error(`Circuit breaker [${this.name}] is OPEN`)
      }
    }
    
    try {
      const result = await operation()
      this.onSuccess()
      return result
    } catch (error) {
      this.onFailure()
      throw error
    }
  }
  
  private onSuccess() {
    this.failures = 0
    this.state = 'closed'
  }
  
  private onFailure() {
    this.failures++
    this.lastFailureTime = Date.now()
    if (this.failures >= this.threshold) {
      this.state = 'open'
      console.error(`[circuit-breaker:${this.name}] OPENED after ${this.failures} failures`)
    }
  }
}

// Usage
const stripeBreaker = new CircuitBreaker(5, 60_000, 'stripe')

await stripeBreaker.execute(() => 
  stripe.paymentIntents.create({ /* ... */ })
)
```

### What the user sees when external services fail

- **Transient failure (caught by retry):** invisible to user, succeeds on retry
- **Persistent failure (circuit open):** graceful error message — "Payments are temporarily unavailable. We've been notified. Please try again in a few minutes, or contact hello@burlingtonconsult.com if urgent."
- **Timeout:** clear message — "This is taking longer than expected. Please wait or try again."

Never show raw error messages. Never show stack traces. Sentry captures the details; the user sees a calm, actionable message.

### Monitoring

- Every retry logged (with attempt count and final outcome)
- Every circuit breaker state change logged as `error` level (triggers Sentry alert)
- Dashboard metric: retries per minute per service (spike = degraded service)
- Dashboard metric: circuit breaker state (any breaker open = investigate)

### Authorization model

Chris is identified by an `is_founder` flag on his profile (only one person has this). He bypasses approval gates.

Other super admins need approval for high-risk changes. Regular admins cannot touch config at all.

```ts
// lib/services/config-authorization.ts
type RiskLevel = 'low' | 'medium' | 'high' | 'critical'

const RISK_RULES: Record<string, RiskLevel> = {
  'pricing.assessment': 'high',              // Changing assessment price
  'pricing.consultation.founder': 'high',    // Founder hourly rate
  'pricing.engagement.standard': 'critical', // Standard engagement fee
  'pricing.engagement.money_back': 'critical',
  'currencies.activation': 'high',           // Turning on GHS, KES, ZAR, etc.
  'providers.stripe.enabled': 'critical',    // Disabling a payment provider
  'providers.paystack.enabled': 'critical',
  'providers.nowpayments.enabled': 'critical',
  'team.deactivate': 'high',                 // Removing a team member
  'team.role_change': 'medium',              // Promoting someone to admin
  'applications.pause': 'medium',            // Pausing new applications
  'features.money_back_guarantee': 'high',   // Disabling a product
  'operations.booking_hold_minutes': 'low',
  'content.homepage_hero': 'low',
  // ...
}

export async function requiresApproval(
  changeKey: string,
  changedBy: Profile,
  changeMagnitude?: { oldValue: number; newValue: number }
): Promise<boolean> {
  // Chris bypasses approval unless the change is critical AND a drop > 50%
  if (changedBy.is_founder) {
    if (changeMagnitude && RISK_RULES[changeKey] === 'critical') {
      const pctChange = (changeMagnitude.newValue - changeMagnitude.oldValue) / changeMagnitude.oldValue
      return pctChange < -0.5  // Even Chris needs confirmation on 50%+ cuts
    }
    return false
  }

  // Other super admins: approval required for medium and above
  const risk = RISK_RULES[changeKey] ?? 'low'
  return risk !== 'low'
}
```

**Execution:** The Railway worker runs a cron job every minute that checks for `scheduled_changes` where `scheduled_for <= now()` and `status = 'pending'`. For each, it executes the change, logs to `config_history`, updates `status = 'executed'`.

4. **Pending drafts** — any content drafts they own must be transferred or discarded.

- Team page card is removed (with a grace period option — "remove from site in 24 hours" for clean transitions)
- Their email is either redirected or disabled (operational decision, not automated)
- All signature instances on the site are removed
- Their historical records stay intact (bookings they delivered, payments that went to them, audit log entries)
- A soft-delete-style marker appears on their records ("Former team member — not current staff")

## Stats & Trust Signals

Burlington is a premium firm competing against both large US immigration firms and cheaper Nigerian agencies. Trust signals are what separate credible from cheap. Every numerical claim on the site — approval rates, client counts, countries served, average times — must be verifiable, sourced, and honest.

### The principle

> If you can't prove it in writing, don't claim it. If you can prove it, cite the source. If the stat changes, update it within 30 days or pull it from the site.

Placeholder stats ("500+ clients served!") on a firm targeting $20,000 engagements destroy credibility faster than having no stats at all. Leading firms quote specific, recent, sourced numbers.

### Database model

```sql
create table public.stats (
  id uuid primary key default gen_random_uuid(),

  label text not null,                       -- Human-readable for display ('EB-1A approval rate, 2025')
  value_numeric numeric,                     -- For numerical claims (percentages, counts)

## Internationalization Stance

**The app is English-only for now. Do not add i18n libraries or `t()` wrappers.**

This is a scope decision, not a positioning one. Burlington serves clients globally (see International by Default section), and the vast majority of EB-1A filers are English-literate — it's the lingua franca of the professional class the firm targets.

Adding internationalization infrastructure before we have translations creates unused abstraction that slows every string change. When we eventually expand to French for Francophone African clients, or Spanish for Latin American clients, we'll revisit. Until then:

- No `next-i18next`, `react-i18next`, `@formatjs/*`
- No `t('some.key')` patterns
- Write strings directly in JSX
- Use British English spelling throughout (matches marketing site, reads as neutral international English)

**What we DO care about is locale-sensitive formatting:**

- Dates formatted per the user's browser locale (`Intl.DateTimeFormat`)
- Currencies formatted per the currency (`$20,000.00` for USD, `₦32,500,000.00` for NGN, `£15,000.00` for GBP)
- Number grouping per the user's browser locale (`1,000` vs `1.000` vs `1 000`)
- Times in the user's browser-detected or stored timezone

These use browser APIs — no extra libraries.

**What we also care about is internationally-neutral copy.** "We did it for ourselves, now we do it for others" (positioning statement) works globally. "As a Nigerian firm, we understand..." positions narrowly. Always choose the first frame.

## Dependency Hygiene

### Before adding a new package

1. **Search the codebase for existing equivalents.** Often we already have something that works.
2. **Check if it's a transitive dep already.** `pnpm why <package>` — if it's there, import from the transitive rather than adding a direct dep.
3. **Check bundle size.** `pnpm dlx bundlephobia <package>` — if > 100 KB gzipped, pause and consider alternatives.
4. **Check maintenance status.** Last commit in the last 12 months? Open issues responded to? Weekly downloads > 10k? If no to any of these, treat with suspicion.
5. **Check license.** Must be MIT, ISC, Apache-2.0, or BSD. Never GPL, AGPL, or SSPL.

### Approval thresholds

- **No approval needed:** small well-known utilities under 50 KB gzipped (e.g., `clsx`, `nanoid`, `slugify`, `zod` if not already installed)
- **Ask Adonai:** anything over 100 KB gzipped, anything with security-sensitive functionality (auth, crypto, parsing), anything that adds a new top-level folder
- **Ask Adonai and Chris:** paid dependencies, dependencies requiring SOC 2 review

### Preferred over rolled-our-own

| Need | Preferred package |
|---|---|
| Validation | `zod` |
| Forms | `react-hook-form` + `@hookform/resolvers` |
| Dates | `date-fns` + `date-fns-tz` |
| Classnames | `clsx` or `class-variance-authority` (cva) |
| IDs | `nanoid` (short, URL-safe) or `crypto.randomUUID()` |

### Never roll our own

## Realtime Messaging & Presence

Premium clients expect messaging that feels like Intercom or Linear — not email. Real-time updates, typing indicators, read receipts, presence. These details signal a firm that takes client experience seriously.

### Technology

**Supabase Realtime** handles this natively. Postgres `LISTEN/NOTIFY` + WebSockets, subscribed to per-row and per-table changes. No Pusher, no Ably, no separate service.

Three things Realtime enables:

1. **Database change subscriptions** — new message arrives → instantly renders in the other party's view
2. **Presence channels** — who's currently viewing a conversation
3. **Broadcast channels** — ephemeral events (typing indicators) that don't need database persistence

### Message delivery

When a message is sent, the flow is:

```
Client A types → sends → Server Action inserts into `messages` table with RLS
                       ↓
                   Supabase Realtime detects INSERT
                       ↓
                   Broadcasts to subscribers of that conversation_id
                       ↓
                   Client B (subscribed) receives the row
                       ↓
                   UI appends message + scrolls + plays subtle sound (if enabled)
                       ↓
                   Client B's view triggers `read_at` update when message scrolls into viewport
                       ↓
                   Client A sees "Read 10:42am" below their message
```

### Schema additions for realtime

```sql
alter table public.messages 
  add column delivered_at timestamptz default now(),
  add column read_at timestamptz;

create index idx_messages_conversation_realtime 
  on public.messages(conversation_id, created_at);

-- Enable realtime on messages table
alter publication supabase_realtime add table public.messages;
```

### Client-side subscription pattern

```tsx
'use client'

import { useEffect, useState } from 'react'
import { createClient } from '@/lib/supabase/client'

export function MessageThread({ conversationId }: { conversationId: string }) {
  const [messages, setMessages] = useState<Message[]>([])
  const [typingUsers, setTypingUsers] = useState<Set<string>>(new Set())
  const supabase = createClient()
  
  useEffect(() => {
    // Initial load
    supabase
      .from('messages')
      .select('*')
      .eq('conversation_id', conversationId)
      .order('created_at', { ascending: true })
      .then(({ data }) => setMessages(data ?? []))
    
    // Subscribe to new messages
    const channel = supabase
      .channel(`messages:${conversationId}`)
      .on(
        'postgres_changes',
        {
          event: 'INSERT',
          schema: 'public',
          table: 'messages',
          filter: `conversation_id=eq.${conversationId}`,
        },
        (payload) => {
          setMessages(prev => [...prev, payload.new as Message])
        }
      )
      .on(
        'postgres_changes',
        {
          event: 'UPDATE',
          schema: 'public',
          table: 'messages',
          filter: `conversation_id=eq.${conversationId}`,
        },
        (payload) => {
          // Handle read_at updates (receipt indicators)
          setMessages(prev => 
            prev.map(m => m.id === payload.new.id ? payload.new as Message : m)
          )
        }
      )
      .on('broadcast', { event: 'typing' }, ({ payload }) => {
        setTypingUsers(prev => new Set(prev).add(payload.userId))
        // Auto-clear after 3 seconds of inactivity
        setTimeout(() => {
          setTypingUsers(prev => {
            const next = new Set(prev)
            next.delete(payload.userId)
            return next
          })
        }, 3000)
      })
      .subscribe()
    
    return () => {
      supabase.removeChannel(channel)
    }
  }, [conversationId])
  
  // Rest of component: render messages, typing indicator, compose box
}
```

### Typing indicators

Typing is ephemeral. Don't write it to the database. Broadcast it.

```tsx
function ComposeBox({ conversationId, userId }: Props) {
  const supabase = createClient()
  const [draft, setDraft] = useState('')
  const channel = useRef<RealtimeChannel>()
  const lastTypingBroadcast = useRef(0)
  
  useEffect(() => {
    channel.current = supabase.channel(`messages:${conversationId}`)
  }, [conversationId])
  
  const handleTyping = (value: string) => {
    setDraft(value)
    
    // Throttle typing broadcasts — don't spam on every keystroke
    const now = Date.now()
    if (now - lastTypingBroadcast.current < 1500) return
    
    channel.current?.send({
      type: 'broadcast',
      event: 'typing',
      payload: { userId, timestamp: now },
    })
    lastTypingBroadcast.current = now
  }
  
  return <textarea value={draft} onChange={e => handleTyping(e.target.value)} />
}
```

**Typing indicator UI rules:**
- Shows as "[Adviser name] is typing..." with animated three-dot ellipsis
- Animation is subtle — bouncing dots at 0.4s intervals, not aggressive
- Disappears after 3 seconds if no new typing event arrives
- Disappears immediately when a message actually sends
- Multiple users typing → "Chris and Afiong are typing..." (cap at 2 names, then "and 2 others")

### Read receipts

When a message scrolls into view, mark it read. This updates `read_at` and the sender sees the receipt update in real time via the UPDATE subscription above.

```tsx
// Use IntersectionObserver to detect when messages come into view
useEffect(() => {
  const observer = new IntersectionObserver(
    (entries) => {
      entries.forEach(entry => {
        if (entry.isIntersecting) {
          const messageId = entry.target.getAttribute('data-message-id')
          const senderId = entry.target.getAttribute('data-sender-id')
          
          // Only mark as read if it's not our own message and not already read
          if (messageId && senderId !== userId) {
            markMessageAsRead(messageId)
          }
        }
      })
    },
    { threshold: 0.5 }  // Message is at least 50% visible
  )
  
  // Observe all message elements
  document.querySelectorAll('[data-message-id]').forEach(el => observer.observe(el))
  
  return () => observer.disconnect()
}, [messages])
```

**Read receipt UI:**
- Sender sees "Sent" (grey check) immediately
- Changes to "Delivered" (grey check) when recipient's client receives it
- Changes to "Read 10:42am" (teal check) when recipient actually views it
- Group chats: "Read by 2 of 3" with hover tooltip showing who

      const lastTimestamp = messages[messages.length - 1]?.created_at
      const { data } = await supabase
        .from('messages')
        .select('*')
        .eq('conversation_id', conversationId)
        .gt('created_at', lastTimestamp ?? '1970-01-01')
        .order('created_at', { ascending: true })
      
      if (data?.length) {
        setMessages(prev => [...prev, ...data])
      }
    }, 5000)  // Poll every 5s when realtime is down
    
    return () => clearInterval(interval)
  }, [isRealtimeConnected, conversationId, messages])
  
  return { messages, isConnected: isRealtimeConnected }
}
```

**Polling rules:**
- Only activate when Realtime connection fails
- Interval 5s by default, 10s if visibility is hidden (tab backgrounded)
- Stop entirely if tab is hidden for >5 minutes (no one's looking)
- Resume Realtime subscription on reconnect, stop polling

### Server Action pattern

```ts
'use server'

import { z } from 'zod'
import { revalidatePath } from 'next/cache'
import { requireUser } from '@/lib/auth'
import { createBooking } from '@/lib/db/bookings'
import { bookingSchema } from '@/lib/schemas/booking'

type ActionResult =
  | { ok: true; bookingId: string }
  | { ok: false; error: string; fieldErrors?: Record<string, string[]> }

export async function submitBookingAction(
  formData: FormData
): Promise<ActionResult> {
  // 1. Auth check
  const user = await requireUser()
  
  // 2. Validate input
  const parsed = bookingSchema.safeParse(Object.fromEntries(formData))
  if (!parsed.success) {
    return {
      ok: false,
      error: 'Invalid input',
      fieldErrors: parsed.error.flatten().fieldErrors,
    }
  }
  
  // 3. Business logic (wrapped)
  try {
    const booking = await createBooking({ ...parsed.data, clientId: user.id })
    revalidatePath('/bookings')
    return { ok: true, bookingId: booking.id }
  } catch (error) {
    console.error('[submitBookingAction]', error)
    return { ok: false, error: 'Could not create booking' }
  }
}
```

### API Route pattern

```ts
// app/api/webhooks/stripe/route.ts
import { NextRequest } from 'next/server'
import { stripe } from '@/lib/services/stripe'
import { handleStripeEvent } from '@/lib/services/stripe-webhook-handler'

export async function POST(req: NextRequest) {
  const signature = req.headers.get('stripe-signature')
  if (!signature) {
    return Response.json({ error: 'Missing signature' }, { status: 400 })
  }
  
  const body = await req.text()
  
  let event
  try {
    event = stripe.webhooks.constructEvent(
      body,
      signature,
      process.env.STRIPE_WEBHOOK_SECRET!
    )
  } catch (error) {
    console.error('[stripe-webhook] Invalid signature', error)
    return Response.json({ error: 'Invalid signature' }, { status: 400 })
  }
  
  try {
    await handleStripeEvent(event)
    return Response.json({ received: true })
  } catch (error) {
    console.error('[stripe-webhook] Handler failed', error, { eventType: event.type })
    return Response.json({ error: 'Handler failed' }, { status: 500 })
  }
}
```

**API route conventions:**
- Always verify signatures on webhooks
- Always log with context (not just "error")
- Return structured JSON errors: `{ error: string, details?: unknown }`
- Use correct HTTP status codes (400 validation, 401 auth, 403 forbidden, 404 not found, 500 server)
- Rate limit public endpoints (via Vercel Edge middleware or Upstash)

### Session & token lifetimes

Configured in Supabase Auth dashboard:

| Token | TTL | Reason |
|---|---|---|
| Access token (JWT) | 1 hour | Short-lived, refreshed automatically |
| Refresh token | 30 days for clients, 7 days for admins | Clients benefit from persistent sessions; admins face elevated risk |
| Password reset token | 60 minutes | Enough time to check email, not enough for stale inbox exposure |
| Magic link token | 15 minutes | Single-use, short window |
| Email verification token | 24 hours | One-time verification at signup |

**Password reset rules:**

- Token expires in 60 minutes
- Token is single-use — invalidated immediately on password change
- All existing reset tokens invalidated when password changes (prevent stale-email replay)
- All existing sessions invalidated when password changes (forces re-login across devices)
- Rate limit: 3 reset requests per email per hour (prevents email bombing)

**Session rules:**

- Sessions tied to session tokens in httpOnly cookies (never localStorage)
- `SameSite=Lax` for portal, `SameSite=Strict` for admin (admin has no cross-site use case)
- `Secure` flag always set in production
- Idle timeout: 24 hours for portal, 8 hours for admin
- Absolute timeout: 30 days for portal, 7 days for admin
- Active sessions visible in `/settings/security` — users can "sign out all devices"

### Error boundaries

Every route segment has `error.tsx`. Every major interactive component is wrapped in an ErrorBoundary with a graceful fallback.

```tsx
// app/(dashboard)/error.tsx — Next.js route error boundary
'use client'

import { useEffect } from 'react'
import * as Sentry from '@sentry/nextjs'

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string }
  reset: () => void
}) {
  useEffect(() => {
    Sentry.captureException(error, { 
      tags: { boundary: 'dashboard-route' },
      extra: { digest: error.digest },
    })
  }, [error])
  
  return (
    <div className="flex min-h-[60vh] flex-col items-center justify-center px-6 text-center">
      <h2 className="font-serif text-2xl">Something went wrong</h2>
      <p className="mt-3 text-sm text-slate-400 max-w-md">
        We've been notified and are looking into it. You can try again, or head back 
        to your dashboard.
      </p>
      <div className="mt-6 flex gap-3">
        <button onClick={reset} className="btn-primary">Try again</button>
        <a href="/dashboard" className="btn-secondary">Dashboard</a>
      </div>
    </div>
  )
}
```

**Where error boundaries are required:**

- Every route segment gets an `error.tsx` (Next.js App Router handles this)
- Every major feature shell — dashboard, booking flow, safelock, documents, messages
- Every third-party integration — Stripe checkout, Paystack checkout, NOWPayments widget, chatbot iframe
- Realtime subscriptions — a failing WebSocket shouldn't white-screen the app

**What goes in a boundary fallback:**

- A clear statement of what happened (in user-friendly terms)
- A "try again" action where it makes sense
- A path forward (back to dashboard, book support, email)
- Never technical details (stack traces, error codes)
- Always captured in Sentry with context

## Testing Strategy

### What to test

| Layer | Tool | Coverage target |
|---|---|---|

### Critical paths — must have tests before launch

These flows touch money, data integrity, or auth. Every one has at least a happy-path Playwright test and a failing-case unit test.

1. **Application submission** — submit valid application → creates profile + application → NDA displayed
2. **NDA signing** — sign NDA → PDF generated → status updates → appears in admin queue
3. **Application approval** — admin approves → client status updates → email sent → onboarding unblocked
4. **Application rejection with reason** — admin rejects with reason → reason captured → email sent with explanation
5. **Consultation booking (Stripe happy path)** — book slot → Stripe checkout → webhook marks paid → booking confirmed → confirmation email
6. **Consultation booking (Paystack happy path)** — same flow via Paystack with NGN pricing
7. **Consultation booking (payment failure)** — checkout fails → booking expires → slot released
8. **Safelock creation** — create safelock → first deposit → balance updates
9. **Safelock completion** — deposits accumulate to target → auto-triggers engagement letter flow
10. **Engagement letter signing** — generate letter → client signs → countersigned PDF delivered → engagement activates
11. **Role-based access control** — client cannot access admin routes; adviser cannot see unassigned clients; super admin can see everything (one test per boundary)
12. **RLS enforcement** — direct DB query as client cannot read another client's data (tested at Supabase level)
13. **Webhook idempotency** — same Stripe event processed twice → no duplicate charge, no double email
14. **Refund flow** — admin issues refund → provider webhook confirms → payment status updates → client notified
15. **Password reset** — request → email → use link → password changes → old sessions invalidated → old reset token invalidated
16. **Admin config change with approval** — admin proposes change → Chris approves → change applies → audit logged
17. **Team deactivation with reassignment** — deactivate team member with active clients → cannot complete without reassignment → after reassignment, deactivation succeeds
18. **Scheduled config change** — schedule price change → worker fires at target time → change applies → audit logged

## For the Human — Working with Claude Code

This section is for **you, Adonai.** How to drive Claude Code effectively through this project.

### 1. Always reference the sprint

When starting a session, frame it like this:

> "We're on Sprint 3, building the booking form. Reference `burlington-webapp-build-plan.md` Sprint 3 for scope. Follow `CLAUDE.md` for code conventions."

Claude Code reads `CLAUDE.md` automatically if it's at the repo root. The sprint reference you provide manually.

### 2. Work feature by feature, not page by page

Good: "Build the booking flow — tier selector, availability picker, checkout trigger. Include the Server Action, Zod schema, and email trigger."

Too vague: "Build the booking page."

Specifying the feature end-to-end prevents Claude Code from forgetting the server-side work.

### 3. Ask for the plan before execution

For anything non-trivial, ask:

> "Before writing the code, walk me through your plan: what files you'll create, what the data flow looks like, what edge cases you've considered."

Review, push back, approve, then let it execute. You'll catch architectural issues cheap here rather than expensive later.

### 4. Keep sessions focused

One feature per session. If you try to do Sprint 3 and Sprint 4 in one go, Claude Code will lose track of state, types, and conventions by the end.

Between sessions, commit progress and start fresh.

### 5. Review the Definition of Done output

Claude Code should output a DoD checklist in its final message. If it didn't, ask for it. If items are marked ✅ but you see problems, push back.

### 6. Run checks before committing

```bash
pnpm typecheck
pnpm lint
pnpm test
pnpm build
```

Don't commit if any fail. Don't let Claude Code convince you failures are "expected" without a clear reason.

### 7. Never deploy without a preview

Every merge to `main` auto-deploys to production. Before merging, verify on the Vercel preview URL. Only merge to `main` after you've tested the preview yourself.

### 8. When Claude Code pushes back, listen

If Claude Code says "this violates rule X in CLAUDE.md, are you sure?" — you're usually wrong. Take the pause. If you still want to proceed, override deliberately and leave a code comment explaining why the rule didn't apply.

## Security Checklist

Before every production deploy, verify:

**Secrets & configuration**
- [ ] All `.env.local` vars set in Vercel dashboard (correct scope: Production / Preview / Development)
- [ ] No `console.log` with sensitive data (grep `console.log` in diff)
- [ ] No hardcoded API keys or tokens (grep `sk_`, `pk_`, known secret prefixes)
- [ ] Gitleaks scan passing on every commit
- [ ] Environment variable Zod validation passing at boot

**Database & auth**
- [ ] RLS policies enabled on every table
- [ ] RLS tested — client cannot read another client's data via direct query
- [ ] Service role key never used in user-facing request paths
- [ ] Auth check at the top of every protected page and Server Action
- [ ] Admin role check at top of every admin page (not just logged-in check)
- [ ] Password reset tokens expire in 60 min, single-use, invalidate on password change
- [ ] Session timeouts: portal 24h idle / 30d absolute, admin 8h idle / 7d absolute
- [ ] Sessions invalidated on password change

**Webhooks & APIs**
- [ ] Webhook signature verification on every webhook endpoint (Stripe, Paystack, NOWPayments)
- [ ] Webhook idempotency via `processed_webhooks` table
- [ ] Rate limiting on all public endpoints (login, signup, apply, chat, assessment)
- [ ] Rate limit 429 responses include `Retry-After` header
- [ ] External API calls have explicit timeouts (no indefinite waits)
- [ ] External API calls implement retry with exponential backoff for idempotent ops
- [ ] Circuit breakers in place for critical external services (payments)
- [ ] CORS policy configured — no `*` with credentials, explicit allowed origins

**Input & output**
- [ ] Input validation (Zod) on every form and API route — client AND server
- [ ] Output sanitization (React auto-escapes, but verify any `dangerouslySetInnerHTML`)
- [ ] File upload limits + MIME type validation via magic bytes (not extension only)
- [ ] CSRF protection (Next.js Server Actions handle this automatically — don't bypass)
- [ ] User-supplied filenames never used as storage paths

**Headers & cookies**
- [ ] Content-Security-Policy header active (not report-only) in production
- [ ] X-Frame-Options: DENY on all responses
- [ ] X-Content-Type-Options: nosniff
- [ ] Strict-Transport-Security with 2-year max-age
- [ ] Referrer-Policy: strict-origin-when-cross-origin
- [ ] Permissions-Policy blocking unused browser APIs
- [ ] Cookies: `httpOnly`, `secure`, `sameSite=lax` (portal) or `strict` (admin)
- [ ] Mozilla Observatory grade: A+ on all three apps

**Monitoring & response**
- [ ] Sentry capturing all error-level logs
- [ ] Health check endpoints on portal, admin, worker
- [ ] External uptime monitoring configured (UptimeRobot or similar)
- [ ] Rate limit spikes alerting Chris
- [ ] Circuit breaker opens alerting Chris
- [ ] Failed webhook deliveries visible in admin dashboard
