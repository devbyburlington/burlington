# Burlington Chatbot API — Integration Spec

**Purpose:** Contract between the Burlington webapp (portal + admin + marketing site) and the internal chat service.

**Audience:** The engineer building the chat service, and the engineer integrating it into the webapp (Ojima).

**Status:** Draft for discussion. Finalize before either side builds integration code.

---

## Context

The chat service is a RAG (Retrieval Augmented Generation) API. It receives user questions, searches a Burlington + USCIS knowledge base, and returns an answer generated by Claude (or Gemini).

Three places consume this API:

1. **Marketing site** (`burlingtonconsult.com`) — floating chat bubble for unauthenticated visitors
2. **Client portal** (`app.burlingtonconsult.com`) — in-app chat for authenticated clients
3. **Admin panel** (`admin.burlingtonconsult.com`) — read-only access to conversation history (does not call the API; reads from Supabase)

The webapp is the source of truth for conversation storage. The chat service is stateless from the webapp's perspective — it receives messages, returns replies, doesn't need to remember prior turns server-side. If conversation memory is needed, the webapp sends prior turns in the request.

---

## Architecture Decision: Server-Side Proxy

The frontend calls the chat service through **our Next.js API route**, not directly from the browser.

```
Browser → app.burlingtonconsult.com/api/chat → Chat Service (Cloud Run) → Claude/Gemini
                   ↓
              Supabase (persist message + reply)
```

**Why:**

- We control auth: our API route verifies the user (if authenticated) before calling the chat service
- We persist to Supabase in the same request, so admin sees conversations immediately
- The chat service's API key never touches the browser
- Rate limiting happens at our edge, before hitting the chat service
- We can add analytics, logging, safety checks without the chat service knowing
- If we swap providers later, the frontend doesn't change

The chat service only needs to accept requests from one origin: our Next.js backend.

## Endpoints

### `POST /chat`

The main endpoint. Single request, single response.

**Request:**

```json
{
  "message": "What's the difference between EB-1A and EB-2 NIW?",
  "conversationId": "uuid-v4-from-supabase",
  "history": [
    { "role": "user", "content": "I'm a software engineer from Nigeria" },
    { "role": "assistant", "content": "Great. For Nigerian professionals..." }
  ],
  "context": {
    "source": "marketing_site",
    "userType": "visitor",
    "userId": null,
    "userEmail": "optional@example.com"
  }
}
```

**Fields:**

| Field | Type | Required | Notes |
|---|---|---|---|
| `message` | string | yes | The user's current question. Max 2000 chars. |
| `conversationId` | string (uuid) | yes | Generated by webapp. Same ID across all turns of one conversation. |
| `history` | array | no | Prior turns in this conversation. Empty for first message. Webapp sends last N turns (we'll start with N=10). |
| `history[].role` | `"user"` \| `"assistant"` | yes if history | Role of the prior turn. |
| `history[].content` | string | yes if history | Content of the prior turn. |
| `context.source` | `"marketing_site"` \| `"portal"` | yes | Where the chat originated. |
| `context.userType` | `"visitor"` \| `"client"` \| `"prospect"` | yes | Authenticated status. `visitor` = marketing site, no account. `prospect` = applied but not approved. `client` = approved client. |
| `context.userId` | string (uuid) \| null | yes | Burlington user ID if authenticated. Null for visitors. |
| `context.userEmail` | string \| null | no | Captured from marketing site visitors if they provided it. |

**Response (success):**

```json
{
  "reply": "EB-1A and EB-2 NIW are both self-petitioned green card pathways...",
  "sources": [
    {
      "title": "USCIS Policy Manual, Volume 6, Part F",
      "excerpt": "The EB-1A requires demonstrating sustained national or international acclaim...",
      "url": "https://www.uscis.gov/policy-manual/volume-6-part-f"
    }
  ],
  "requiresEscalation": false,
  "escalationReason": null,
  "conversationId": "uuid-v4-echoed-back",
  "messageId": "uuid-v4-for-this-reply",
  "model": "claude-haiku-4-5",
  "latencyMs": 1240
}
```

**Fields:**

| Field | Type | Notes |
|---|---|---|
| `reply` | string | The assistant's response to show the user. Plain text or markdown — specify which. |
| `sources` | array | Knowledge base passages used to generate the reply. Can be empty. |
| `sources[].title` | string | Human-readable source title. Shown in UI. |
| `sources[].excerpt` | string | Short excerpt (< 300 chars). Shown in UI as "based on..." reference. |
| `sources[].url` | string \| null | Optional external link. |
| `requiresEscalation` | boolean | True if the chat service couldn't confidently answer and recommends human follow-up. |
| `escalationReason` | string \| null | If escalation needed, why. E.g., "out_of_scope", "low_confidence", "sensitive_topic", "user_requested_human". |
| `conversationId` | string (uuid) | Echoed back. Confirms continuity. |
| `messageId` | string (uuid) | Unique ID for this assistant message. Webapp uses this to store the reply. |
| `model` | string | Which LLM generated this reply. Useful for analytics. E.g., `"claude-haiku-4-5"`, `"gemini-1.5-flash"`, `"claude-sonnet-4-6"`. |
| `latencyMs` | number | Time taken by the chat service to generate the reply. Operational metric. |

**Response (error):**

```json
{
  "error": {
    "code": "rate_limited",
    "message": "Too many requests. Try again in 30 seconds.",
    "retryAfterMs": 30000
  }
}
```

| Code | Meaning | HTTP status |
|---|---|---|
| `invalid_request` | Malformed payload, missing fields | 400 |
| `unauthorized` | Missing or invalid API key | 401 |
| `rate_limited` | Too many requests | 429 |
| `model_unavailable` | Upstream LLM is down | 503 |
| `knowledge_base_unavailable` | RAG index is down | 503 |
| `internal_error` | Anything else | 500 |

### `GET /health`

Health check endpoint. Used by webapp monitoring + uptime checks.

**Response:**

```json
{
  "status": "healthy",
  "version": "1.2.0",
  "checks": {
    "knowledgeBase": "ok",
    "llm": "ok"
  }
}
```

Return `503` if any check fails.

### Streaming (optional — v2)

For better UX, streaming the reply token-by-token is a significant upgrade. Not required for v1, but worth planning.

**If supported:** same `/chat` endpoint with header `Accept: text/event-stream`. Response is Server-Sent Events (SSE):

```
event: content
data: {"delta": "EB-1A"}

event: content
data: {"delta": " and EB-2 NIW"}

event: sources
data: {"sources": [...]}

event: done
data: {"messageId": "uuid", "model": "claude-haiku-4-5", "latencyMs": 1240}
```

Webapp proxy forwards SSE to the browser. Adds ~50ms latency but massively improves perceived speed.

**Decision:** SSE support is v2. v1 ships with non-streaming.

## Authentication

The webapp sends an API key in the `Authorization` header:

```
Authorization: Bearer <BURLINGTON_CHAT_API_KEY>
```

**Rules:**

- One API key per environment (dev, staging, production)
- Keys are rotatable on request
- Keys never leave server-side code (Next.js API routes) or `.env` files
- Keys are stored in Vercel environment variables (scoped to the right environment)
- The chat service should reject requests from unknown keys with `401`
- Future: consider IP allowlisting Vercel's egress IPs (belt and suspenders)

---

## Rate Limiting

The chat service should rate limit per API key to prevent runaway costs.

**Suggested limits:**

- Per API key: 60 requests/minute sustained, burst to 120
- Per conversation ID: 20 requests/minute (prevents a single user hammering)
- On 429 response, include `retryAfterMs` in error body

The webapp will also rate limit at its own edge (per IP for marketing site, per user for portal) before the request reaches the chat service.

## Conversation Storage — Where It Lives

**Webapp side (our Supabase):**

All conversations and messages. Source of truth. Drives the admin inbox.

```sql
-- Already in our schema
chat_conversations (id, client_id, source, visitor_email, status, assigned_to, ...)
messages (id, conversation_id, sender_id, sender_type, content, ...)
```

When a user sends a message:
1. Webapp inserts the user message into `messages`
2. Webapp calls chat service with the full payload
3. Webapp inserts the assistant's reply into `messages` (using `messageId` from the response)
4. If `requiresEscalation=true`, webapp flags the conversation as `escalated` and notifies admin

**Chat service side:**

May log requests/responses for:
- Debugging
- Model evaluation
- RAG performance tuning
- Usage analytics

These logs are operational, not the source of truth. Retention policy: at your discretion, but recommend 30–90 days then purge. Do NOT log PII (userEmail, userId) in plain text — hash or redact.

## Escalation to Human

The chat service returns `requiresEscalation: true` when:

- Confidence in the reply is below a threshold
- The user explicitly asks for a human ("I want to talk to someone", "Connect me to a real person")
- The topic is outside scope (legal advice requests, complex individual case analysis)
- The topic is sensitive (the user mentions emergency situations, distress)

**What the webapp does when it sees `requiresEscalation: true`:**

1. Still shows the chat service's reply (it may contain useful context)
2. Appends a UI element: *"Would you like to book a consultation or speak with our team?"*
3. Flags `chat_conversations.status` as `'escalated'` in Supabase
4. Sends a notification to the admin team (email + in-app)
5. In the admin chatbot inbox, conversation appears at the top of the escalation queue

The chat service does NOT directly notify the admin team. The webapp mediates.

## Model Selection

Chat service handles model selection internally. Webapp doesn't need to specify.

**Suggested logic (chat service's decision):**

- Simple FAQ / definition queries — Haiku (fast, cheap)
- Complex strategy / eligibility questions — Sonnet (more reasoning)
- Triage: start with Haiku, retry with Sonnet if confidence is low

The model used is returned in the response (`model` field) so webapp can track costs and quality by model.

## Testing & Launch

**Before production launch:**

1. Chat service team provides a working staging endpoint
2. Webapp integrates against staging
3. Joint test: 50 test queries covering the FAQ range, edge cases, escalation scenarios
4. Verify conversation records appear in Supabase correctly
5. Verify admin inbox shows escalated conversations
6. Load test: 100 concurrent users for 5 minutes
7. Verify rate limiting works as specified
8. Verify graceful degradation when chat service is down (fallback UI: "Our assistant is temporarily unavailable. Please book a consultation or email us.")

## Open Questions (to resolve before implementation starts)

1. **Domain structure.** What subdomain for the chat service? `chat.burlingtonconsult.com` vs `chat-api.burlingtonconsult.com` vs something else. DNS setup needed either way.

2. **Response format: markdown or plain text?** Clients will see the reply rendered — if markdown, we need a markdown renderer on the frontend. If plain text, chat service should avoid markdown syntax in responses.

3. **Conversation history window.** How many prior turns should the webapp send? Too few = loses context. Too many = high token cost. Starting suggestion: last 10 turns. Revisit after launch.

4. **Source citations UI.** If the chat service returns `sources`, how prominent should they be? Inline with the reply, collapsed below, tooltip on hover?

5. **Escalation notification channels.** Email only, or also Slack / WhatsApp for faster admin response?

6. **Fallback behaviour.** When chat service is unavailable (503), what does the user see? Options: generic error message, static fallback response, direct them to book a consultation.

7. **Analytics.** What events should the webapp track? Every message? Only sessions? Conversation length, escalation rate, user satisfaction (thumbs up/down)?

8. **Moderation.** Does the chat service do any content moderation on the user's input (PII detection, prompt injection defense, abuse filtering)? If not, should the webapp?

9. **Multi-language.** Does the chat service support non-English queries? Nigerian clients may write in Pidgin or mix languages. Should the knowledge base include non-English content?

10. **Conversation continuity across sessions.** If a visitor closes the tab and returns a week later, do they see their prior conversation? If yes, how is the `conversationId` recovered — cookie, email lookup, user account?

## Contract Summary

**Webapp responsibilities:**

- Call `POST /chat` with correct payload
- Persist all messages to Supabase
- Authenticate users (portal) and identify visitor context (marketing site)
- Rate limit at its own edge
- Handle all errors gracefully with user-friendly messages
- Store and surface escalated conversations to admin
- Never call chat service from browser — always server-side proxy

**Chat service responsibilities:**

- Accept requests matching the spec above
- Return responses matching the spec above
- Run RAG against the knowledge base
- Call LLM (Claude or Gemini) and return reply
- Detect when human escalation is needed
- Rate limit per API key
- Return health status on `/health`
- Keep staging + production environments deployable and healthy
- Log operationally but don't store PII in plain text

**Joint responsibilities:**

- Agree on this spec before either side builds integration
- Define knowledge base ingestion process for new Burlington content
- Monitor costs and usage jointly
- Coordinate on model changes, prompt updates that might affect reply quality
